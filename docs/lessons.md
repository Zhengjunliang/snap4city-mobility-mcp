# Lessons — 架构陷阱沉淀

> 每条 `L{n} {仍生效的绑定事实 + 修复}`. 只留仍约束当前设计的坑。

---

L18 **LangGraph 节点返回的键必须在 State schema 里有通道, 否则被静默丢弃** —— 设计 tpl missing-slot 路径时差点踩中: `execute` 想返回 `{"missing": [...]}` 给 `respond`, 但 `AdvisorState` (TypedDict) 没有 `missing` 通道 → LangGraph 对 schema 外的节点输出键**静默丢弃** (版本不同也可能直接报错), `respond` 永远收不到, 且测试 mock 单测节点函数时不会暴露 (只有走 `graph.ainvoke` 全图才丢)。修复策略: **派生数据不进 state** — `respond` 端从已有通道重推导 (missing slots 由 `_missing_slots(intent, slots)` 从 `slots` 通道现算, 必需 slot 表 `tpl.REQUIRED_SLOTS` 单一来源供 `run_tpl_flow` 跳链和 respond 提问共用)。要新增跨节点数据, 要么进已有通道 (如塞进 `slots`/`tool_results`), 要么显式扩 `AdvisorState` 字段 — 禁止"返回了就当传到了"。

L17 **km4city geocoder 把模糊 POI 排在真地点前 — geocode 两段式 + 选点按 label 匹配** —— `excludePOI=false` (L11 为找地标所设) 让 POI/目录命中排到真广场前面: "Piazza Duomo" 第 1 个 feature 是公司 `PIAZZA DUOMO DI PRIZIO STEFANO & C. S.A.S.` (43.7736, 11.2421, 离真 Duomo 偏西 1.1 km) → 步行路线 1.83 km vs 基准 0.68 km; "piazza Dalmazia" 第 1 个是 address:null POI, 而 address=`PIAZZA DALMAZIA` 的精确地址条目排第 4。修复两层: (a) [mcp_tools._geocode_address_first](../src/snap4city_mobility_mcp/mcp_tools.py): 先 `excludePOI=true` 查 (纯地址条目, 坐标在可路由街图上), Tuscany 区内空/异常才退 `excludePOI=false` 兜底 (车站/地标只存在于 POI 目录, L11 仍成立) — 任一查询都不比单段差; (b) [orchestrator._pick_coord](../src/snap4city_mobility_mcp/orchestrator.py): 不再盲取 `features[0]`, 优先第一个 address/name 分词 (casefold + NFKD 去重音 + 去意语虚词 di/del/della/…) **⊆** search 分词的 feature — 子集方向故意严格, 带额外 token 的标签被拒 (上述公司名含 "piazza duomo" 但有多余词, 不会误中), 无匹配退回 server 排序第一个。顺带: geocode debug 日志必须带 picked feature 的 index+address — 这次诊断就是吃了日志只有坐标没 address 的亏。另注: geocoder 索引比 L11 已知的更杂, "via dello Steccuto" (真实佛市街道) 返回 100 条全荷兰 Maastricht 命中、零托斯卡纳。**二轮坑 (同日实测)**: 地址 pass 又暴露**跨城同名** — "Piazza Duomo" 首位是 CASTELNUOVO DI GARFAGNANA / PIETRASANTA 的同名广场 (90 km 外), "Santa Croce" 全是 SANTA CROCE SULL'ARNO 的 VIA BENEDETTO CROCE → 步行 routing 必挂, 每次还白烧 ~54 s 重试梯子。补修: (c) [mcp_tools._narrow_by_city](../src/snap4city_mobility_mcp/mcp_tools.py) 城市阶梯 — 用户点名的城市 (city 分词 ⊆ search 分词) > FIRENZE 默认 > 全托斯卡纳, 两个 pass 都先要城市置信命中, 都没有才退全托斯卡纳地址命中; (d) UNDERSTAND_SYSTEM 要求把用户点名的城市留在 place text 里 ("in Firenze" → "piazza Duomo, Firenze"), 禁自行编造城市; (e) 延迟: foot 兜底 profile 改单次探测 (`attempts=1`) — 请求的 profile 已烧完整 L3 梯子 (~27 s), transient 已排除, 兜底再跑 3 次纯属浪费 (~省 22 s)。

L16 **会话级对象别在 `run_advisor` 里每轮重建** —— chat.py 实测每个回合都打一遍 `[INIT]/[LOAD_TOKEN]` 噪音并重拉 `/apps.json`: 根因是 `run_advisor` 每次调用都 `_build_config()` + `Llama4Client()` (= 新 TokenManager), 还顺带每轮 `fetch_tool_schemas` (确定性图后 schema 已无人消费, 纯多一个 round trip)。token 本体早已持久化在 `token_stored.json` (不会重登录), 浪费的是对象重建 + 网络往返 + stderr 噪音。修复: orchestrator 模块级懒缓存 `_CFG`/`_LLM` (进程生命周期, token 过期刷新本就在 `TokenManager.get_token()` 内部按需做, 无 stale 风险); 删每轮 schema 拉取; TokenManager 命中有效 token 的热路径静默。MCP `Client` 刻意仍每轮 `async with` 重连 (生命周期干净, 内网握手便宜)。诊断走 `logging.getLogger("snap4city_mobility_mcp")` → chat.py 挂 FileHandler 写 `debug.log` (gitignored), 不污染 widget JSON 形状。

L15 **JupyterHub 上 `pip install` 绝不进 base env** —— 否则一条命令能让整个 singleuser server 起不来。踩坑实录: 为给 Chainlit 配 `jupyter-server-proxy`, 用 base 的 `/opt/conda/bin/pip install jupyter-server-proxy` (proxy 4.x 要 `jupyter-server>=1.24`) → 把 base 的 `jupyter-server` 从 **1.11 强升到 2.18**, 连带 `jupyter-client/jupyter-core/nbformat/traitlets` 全升, 撞坏 `jupyterlab 3.1` / `nbclassic` / `ipykernel 6.4` (这些钉死老版本)。当时不报致命错、server 还在跑 (改的是磁盘, 内存进程没动) → **一旦 Stop/Start server 重新加载就崩**: `Spawn failed: didn't respond in 30 seconds`, 反复重试、新建 named server 都一样。根因: 容器 overlay 在 stop/start 间**被复用** (非 fresh), 坏掉的 `/opt/conda` 持久 → 自愈无门; 且没 server 就没终端, 改不了 `/opt/conda` → 只能让管理员 (referente) `docker rm` 容器重建。代价惨重 (丢一个工作会话)。**铁律**: (a) 一切 `pip install` 只在专用 conda env (`s4c`) 内, `conda activate s4c` 后再装, 永不碰 base 的 jupyter 包; (b) 装前看依赖约束, proxy 4.x 这种会强升 jupyter-server 的别在老镜像 (jupyter-server 1.11/lab 3.1) 上装; (c) 容器临时层 (overlay) 上的改动 stop/start 复用会保留, 唯有 fresh spawn (referente 重建) 才清。代码安全靠持久卷 `/home/jovyan` (`/dev/sda1`), 与 `/opt/conda` (overlay) 分离。

L14 多轮对话状态**早已存在**于对外输出 JSON 的 `messages` (OpenAI 格式, `run_advisor(query, history)→{...,messages}`) —— 想加聊天前端时别"从头造多轮", 它只是把上轮 `messages` 回传当 `history` 的纯胶水。陷阱1 (平行实现): 加新聊天入口时若**留着**旧的, 就成两份多轮实现, 违单一正确实现 (CLAUDE.md §2 规则7) → 同一时刻只留一个聊天入口。陷阱2 (选错形态): 一开始选了 Chainlit (web UI), 但唯一运行环境是锁死的 JupyterHub, 暴露 web 端口要 `jupyter-server-proxy` + 子路径 `--root-path` + server 重启, 极脆 (并直接触发 L15 灾难) → **改用终端 `chat.py` REPL** (`python chat.py`, 纯终端、无 web、无 proxy, 在 JupyterHub 一跑就通)。两个落地点: (a) `run_advisor` 本就 async → `chat.py` 用 `asyncio.run(main())` 在 `while` 循环里 `await`; (b) **只显 LLM 原话** `messages[-1].content` (`respond` 已把距离/ETA 措辞进自然语言并兜底), **不硬编码** `📍km·ETA`; 全量 widget JSON (含 `data.wkt`) 每轮追加 `outputs.txt` 供离线看流程。教训: 选前端形态先问"运行环境能不能跑", 锁死的 JupyterHub 优先终端而非 web。

L13 Llama4 `tool_choice="auto"` 自由调工具**不可靠**: 模型一旦想叙述 (如 routing 失败后写 "retrying with a different profile…") 就把 tool call 写成 **prosa 前缀 + pythonic 文本** `The request failed…[routing(...)]` 而非结构化 `tool_calls` → 漏成最终答案, `data` 空, 工具没真跑 (是 L11 同类问题反复发作; `recover_pythonic_tool_calls` 打补丁只治标, 且 prosa-前缀变体它也接不住)。**治本: 砍掉 `agent ⇄ tools` agentic 回路**, 改线性确定性图 `understand → execute → respond`: `understand` 用 forced `tool_choice={extract_slots}` (强制结构化, 从不走 pythonic) 只提取 `{intent, origin, destination, mode}`; `execute` 纯 Python 按 mode 跑固定流 (geocode×2 + `routing(mode)`, foot_quiet 失败确定性回退 foot_shortest); `respond` 用 `tool_choice="none"` (不传 tools) 只措辞 → LLM **永不自由调工具**, pythonic-leak 这条路彻底消失。代价: tpl_* 发现链暂不支持 (返友好提示)。`recover_pythonic_tool_calls`/`_parse_pythonic_calls` 一族 + agentic 节点已删 (单一正确实现, 历史归 git)。

L2 km4city `routing` 的 `journey.routes` 是 **list** of `{arc,distance,eta,time,wkt}` (OpenAPI spec 误写成 dict); `ok` / HTTP 200 ≠ 找到路 — `routes` 可空 (车在步行街 / 起终点同点)。必检 `len(journey.routes) > 0`; 写工具前 live-probe, 别只信 spec; 整 envelope 透传上层按真实结构挑。

L3 km4city 冷启动 / 短时间重复命中相同坐标对时偶发返**临时空 routes**, 几秒后自愈。`routing_with_retry` 隔 6s 重试至多 2 次 (共 3 次) 遮掉; 第 3 次仍空按 L8 处理 (原始 payload 进 debug.log 供判别)。transient — 跟稳定的 L8 区分。

L4 加 / 改 `[project.scripts]` 后, `.venv` 启动器 stub **不会**因源码改动自动重生 — 跑 `uv run <name>` (自动 sync + 重建) 或 `uv pip install -e .`。(`'<name>' is not recognized` ≠ PATH 问题。)

L5 dashboard `/apps.json` 返回的 server URL **写死内网 IP** (`http://192.168.1.117:8000/...`), host 不同就连不上。[mcp_tools._build_config](../src/snap4city_mobility_mcp/mcp_tools.py) 拿到 cfg 后先把该 IP 替换成 `DASHBOARD_URL` 再喂 `Client`。

L6 FastMCP **只在 multi-server cfg** 才给 tool 名加 `<server>_` 前缀; 我们 single-native-server cfg 不加, 直接 `call_tool("routing", ...)` 裸名即通。扩到 multi 会突然加前缀, 所有 hardcoded 名崩 — `_build_config` 只挑 native 保持 single-server。

L7 km4city `routing` envelope 里 `response.error_code` 是**唯一**成功权威 (`"0"` = OK); `response.error_message` 成功时也非空 (`"successful"`)。永远别拿 `error_message` 当失败信号。

L8 referente 的**车在中心步行街** (Florence ZTL Duomo→Santa Croce car) 返裸顶层 `{"error": ""}` (wrapper bug, 没透传 km4city `-2` envelope); **稳定**, 重试不愈 (不同于 transient L3), 表现成 "empty body" → 建议改 foot/public_transport。第 3 次仍空 = L8; 隔 ≥5s 出 envelope = L3。**foot 模式也实测出现过** (2026-06 Rifredi→Morgagni foot_shortest 空 body 重试不愈) — 不只 car-ZTL; 步行档现自动互换 profile 重试 (`_FOOT_FALLBACK` 双向), 仍空则凭 debug.log 的原始 payload 报 referente。

L9 JupyterHub 默认 kernel = Python 3.9, 太老装不了 fastmcp (要 ≥3.10), `pip install fastmcp` 报误导性的 "no matching distribution"。建 conda 3.11 env + ipykernel (`s4c`)。git 可用, uv 通常没装 (用 pip/conda)。

L12 Llama4 后端 (vLLM) context 一大就**炸/幻觉**: 网关返 HTTP **200** 但 body 包上游错误 `Failed to make POST request to ...:8080/serve/llama4-agentic-inference. Error: 500 Server Error: Internal Server Error` → 原 `_is_transient` 不认 (只列 502/503/504) → 不重试直接崩。根因常是请求太大: `tools` 节点把工具原始返回**整个塞回** messages (geocode 100 个 feature / routing 超长 wkt+逐 arc), 每轮重读越堆越大。修: (a) `_TRANSIENT_HINTS` 加 `internal server error`/`server error`/`failed to make post request` → 当 transient 重试; (b) [mcp_tools.slim_result_for_llm](../src/snap4city_mobility_mcp/mcp_tools.py): 喂模型的 **message** 瘦身 (geocode 留 top-5 + {address,city,coordinates}; routing 丢 wkt, arc→去重街道名), 而 **audit `tool_results` 留完整** → widget `data` 不受影响 ([orchestrator.tools](../src/snap4city_mobility_mcp/orchestrator.py) 分两路)。小 context = 少 500 + 少幻觉。

L11 km4city geocoder **不再 region-locked** (索引现含 Valencia/ES + 法国南部), `address_search_location("...Firenze")` 可返 100 条全西班牙/法国命中、零托斯卡纳。叠加 `excludePOI` 默认 `true` 把"Piazza del Duomo"(广场=POI)滤掉 → 只剩模糊街道匹配落到西班牙。schema 无地理约束参数。修复 ([mcp_tools.exec_tool](../src/snap4city_mobility_mcp/mcp_tools.py)): 强制 `excludePOI=false` (让地标可被找到) + 客户端按 `TUSCANY_BBOX` (lng 9.6–12.5, lat 42.2–44.5) 过滤 feature, 保留 score 序; 区内空 → 返 `{error}` 给 agent 重试。另: Llama4 偶把 tool call 写成裸 pythonic 文本而非结构化 `tool_calls`, 两种变体都会漏成最终答案 → `data` 空、routing 不真跑、答案是幻觉: (a) `;` 分隔多调用 `fn(); fn()`; (b) `[fn(...)]` 后粘聊天模板 `assistant\n\n<编造答案>`. `_parse_pythonic_calls` 已加: exec-mode 解析多语句 + `_leading_bracket_group` 只取开头平衡 `[...]` 丢尾部模板文本。**(geocode 过滤部分仍生效; "强制 excludePOI=false" 已被 L17 两段式取代 — 地址优先、POI 兜底; pythonic 恢复部分已被 L13 取代 — agentic 回路与 `_parse_pythonic_calls` 一族删除, LLM 不再自由调工具。)**

L10 Llama4 旧 endpoint `llama4-inference` 即使 auth 成功 (200) 也返 `{"message":"Rule not found..."}` (~0.1-0.3s) — auth ≠ inference 授权。用 OpenAI 兼容的 agentic endpoint `llama4-agentic-inference` ([llm.py](../src/snap4city_mobility_mcp/llm.py) 默认), 顺带解锁原生 tool calling。响应是 OpenAI `{choices:[{message:{content,tool_calls}}]}`。
