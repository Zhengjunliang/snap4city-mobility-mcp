# Lessons — 架构陷阱沉淀

L59 **迁移: mobility MCP server 托管到 Snap4City server (固定 URL, 非 apps.json), JupyterHub 只跑桥** —— 我们的 `mcp_server.py` (geocode + `route`) 被部署到 Snap4City server, 直连 `http://192.168.1.117:8000/snap4agentic-mobility-advisor/mcp` (probe 确认暴露 `address_search_location` + `route` 两个裸名工具)。改动: `LOCAL_MCP_URL` 默认从 `DASHBOARD_URL` 派生 (`f"{DASHBOARD_URL}/snap4agentic-mobility-advisor/mcp"`) → **一个 `S4C_DASHBOARD_URL` 同时挪 native + mobility 两个端点到同一 host**, 行为默认不变; 本地自测仍可 `export S4C_LOCAL_MCP_URL=http://127.0.0.1:8020/mcp` 起 :8020 覆盖。JupyterHub 生产链路**只剩桥一个进程** (`uvicorn api:app :8010`), 不再手启 :8020; 改 `mcp_server.py` 要 Snap4City 那边重部署 (漏这步 = 跑旧工具, 见 [[L54]])。**两 client 拆分保留** (mobility 与 native 是两个不同 MCP server, 各单 server 保裸名避 [[L6]] 前缀税, 合并要 prefix-aware 重构不值)。**文档措辞约定**: 对外文档 (README/relazione/docs/UML) 托管归属**一律 "Snap4City server", 不用 "referente"** (内部代称; CLAUDE.md/lessons.md 等不发布的内部文件里保留)。app 半边 (orchestrator+桥+Llama4) 归属仍未定 (见 memory [[project-server-deployment-migration]] — B 路线的 Llama4 来源 IP 授权是门槛)。教训: 部署收口先确认"托管什么" (工具型 MCP server vs 带 LLM 的 app); 让**单一 env 派生移动多个端点**是干净的收口招, 且默认值保持不变 = 零行为回归。

L58 **纯附近搜索 (pins-only, 无 route): 地图不能靠撒 pin 自居中, 且"追问轮"绝不能带 pin 数据否则清图** —— 新 `nearby` intent (用户 "mostrami le farmacie qui intorno" 只要一片 pin 不要线) **复用现有 slot** (`destination_category` 载类别 + `origin_text` 载中心地址/città, **不新增 slot**; `request_type` enum 加 `nearby`, `_request_to_intent` 映射), execute 单近搜 `maxresults=NEARBY_MAX=50` 出 `[{name,lat,lng,uri,distance_km}]` 挂**顶层 `data.services`** (与 route 的**嵌套** `routes[i].services` 区分, 前端 `else if (Array.isArray(data.services))` 据此判别 pins-only 轮次)。三个陷阱: (1) **widgetMap 只在 `addCustomTrajectory` 隐式 fit-bounds 时居中, `addSelectorPin` 单发不移动/缩放地图** → pins-only 无内置居中手段 (官方 center 事件未经 referente 确认, 规则 8), 决策 = **不居中** (地图停原处, 已接受); (2) respond **只在真正搜索过 (非 unsupported) 才设 `data.services`** —— 缺类别/缺地点的追问轮若也带 `data.services=[]`, 前端会**清空地图**, 与 route 追问 (data={} 不动图) 不一致; nearby 回复走 `_compose_reply` **早于** route 的 `if not routes` 失败分支 (空 routes 不再误判为失败), 计数用单/复数 ("C'è 1 farmacia" / "Ci sono N farmacie", `_bare_plural` 剥 `_category_it` 复数的冠词); (3) 新跨节点数据 `nearby` 必须加 `AdvisorState` channel (见 [[L18]])。另: 老师提的 "LLM 超时 60→120" 代码里 `LLM_TIMEOUT_S` **本就是 120** (git born at 120, 从无 60), bus 超时亦已 120 —— **改前先验证前提**, 别照"要改的数字"凭空找目标。

L57 **respond 阶段的 LLM 可整段移除: 它的输出全部可由 execute 的确定性数据推导, 保留只是给网关多挨一次 504** —— respond 只把已算好的距离/时长/地名/pins 包装成人话, 是唯一**非承重**的 LLM 调用 (understand 的 NL→slot 抽取不可去)。修: 删 `RESPOND_SYSTEM` + 只喂 prompt 的中间视图, `_compose_reply` 纯 Python 出意语措辞 (覆盖 missing/unsupported/单+多模式含最快/departure/GPS 起点/类别目的地/沿途服务/pt_degrade), 加 `_CATEGORY_IT` 英文类别→意语表。每轮 LLM 调用 2→1, 顺带消除语言漂移与编造风险; widget JSON 对外形状不变 → 前端 + api.py 零改动。原则: 确定性数据的"措辞层"别默认塞 LLM, 每次 LLM 往返都是一次网关 504 敞口。

L56 **固定半径 near-search 空手 = 无报错、无日志、静默丢整个特性** —— car 轮次一切正常唯独 parking pins 不见: `Car_park` 近搜在固定 0.5km 内真的没有停车场 (近郊常态), 返空 FeatureCollection 无 error。修: `PARKING_RADII_KM=(0.5, 1.0)` 空 rung 扩半径重搜 (复用"只有 deciding call 进 audit"约定)。**阶梯上限按特性语义定**, 不照抄目的地解析的 10km (停车场离目的地太远, 步行就没意义)。"为什么没 pin"先看 debug.log 的 near-search 返回行。

L55 **km4city civic 坐标锚在地块角、同侧多号成簇堆点: civic 阶梯选号正确 ≠ 落点精确** —— "via santa marta 3" 精确命中 `civic='3'` 但 pin 偏 ~165m; probe 实锤数据本身: 该街奇数号 1/3/5 **共用同一坐标** (地块角), 偶数侧反而准。SSM 连 civic 条目都没有 → 生态内无更好门牌数据。结论: **不改代码** (civic 阶梯行为正确, 见 [[L52]]), 偏差落在同一条街上可忽略; demo 用 POI 名 ("dalla facoltà di ingegneria santa marta") 走 POI pass 落真实建筑。排查端点漂移分两层: 先证代码选对了候选 (log `civic=`), 再 probe 数据源看候选本身准不准 —— 多个相邻 civic 坐标完全相同 = 数据占位符的指纹。

L54 **LLM 网关延迟是分钟级波动的部署期变量: 等待窗按实测尾延迟设, LLM 输入侧必须有界** —— understand 一次 slot 抽取实测可达 67s (基准 20-30s, 端点侧波动归 referente), 叠加 bus 与沿途服务后超过前端轮询上限 → 把"慢"误报成"挂"。修三处: `POLL_MAX_MS` 420s (按实测最坏轮次求和); understand 的 convo 裁到**最近 8 条**; chips 选择**只写一句 user 回声进 history** (整块 detail 站表不再喂进后续每轮 prompt)。原则: 能用一句结构化文本承载的语境, 别用几百 token 的展示文本承载。`llm.py` 两个 transient 分支各记 WARNING (失败尝试此前零日志)。**另: 本地自测时 mcp_server (:8020) 与桥是两个进程, 都要重启** —— 漏重启会伪装成离奇 bug (曾表现为 civic 全 None 端点漂移), 别急着改代码先核对进程版本 (mobility server 现托管在 Snap4City server; 生产链路 JupyterHub 只有桥一个进程, 改 `mcp_server.py` 要 Snap4City 重部署)。

L53 **沿途服务 = 点半径 API 客户端造走廊** —— 远程 `service_search_near_gps_position` 只支持点+半径 (无 corridor 参数) → `_sample_polyline` 沿 route wkt 每 0.4km 采样 (半径 0.25km ≈ 间距一半, 圆盘盖满线不叠), `_service_anchors` 按 mode 分规则 (bus leg 只首尾顶点 = 上下车站, foot leg 照采), 锚点去重 cap 8, 并发近搜后按 uri 去重取最小锚距 cap 10/mode。新 slot `services_category` (与 `destination_category` 语义区分: 沿途想看的 vs 目的地本身), 只在用户点名时触发。三个设计陷阱: (1) **anchor 调用不进 audit** —— 审计里出现一串 Pharmacy 条目会让下游把沿途药店讲成终点; (2) 服务**按 mode 挂 `routes[i].services`** (新跨节点数据必须显式加 AdvisorState channel, 见 [[L18]]), 前端 chips 切换 pin 集合纯本地; (3) **pin 图标不用管** —— `addSelectorPin` 的 query 带 `serviceUri`, widget 自己 resolve 并画官方类别 icon, 我方只定底色区分集合。教训: 点半径 API 造走廊, 间距 ≈ 2×半径是覆盖/开销平衡点, 锚点与结果都要 cap; 平台能按 uri 自渲染实体 icon 时别自建"类别→图标"映射表。

L52 **geocode 门牌精度双杀: normalization 丢弃判别字段 + anchor-nearest 覆盖 score 序; widget pin 删不掉 = remove 事件缺 layer key** —— "via laura 11" 曾选中名叫 "LAURA" 的 POI。两层根因: `/location/` 明明把 `serviceType:"StreetNumber"` + `civic` 精确门牌排第一, 但 `_normalize_feature` 把这两个字段**丢弃** → 同街候选字节级不可分; 而 `_pick_feature` 的 anchor-nearest ([[L43]]) **无条件覆盖** server score 序, 终点恒有锚 → civic 精确命中永远赢不了。修: normalization 透传两字段; `_pick_feature` 加 civic 阶梯 —— 搜索文本含门牌号 (`_house_number`: 最后一个 1-4 位独立数字, 5 位邮编排除) 时先收窄 civic-exact, 无则收窄街型标签 (`_ROAD_WORDS`, 治 "LAURA" POI), 每级只在命中时收窄 → **无号查询逐字节不变**。**Pin 侧**: `removeSelectorPin` 的 handler 按 `passedData.desc` 查 `gisLayersOnMap[desc]` 删 layer, 我们只发 `{query, queryType}` 缺 desc → 查空**静默 no-op**, pin 永久累积。修: add 时记录 `{desc, query}`, remove 原样回发。教训: 排序/选点 bug 先查 normalization 层丢了什么 (服务端早给对了, 客户端自己扔掉了判别字段); 对第三方事件 API, "事件名对"不够 —— **key 字段错照样静默失败**, 落地前先读上游 handler 源码。

L51 **bus 乘车段真实走线 = 运行期调公开 km4city tpl API 换 shape; 跨 ingest 只有几何是通用键** —— [gtfs_shapes.py](../src/snap4city_mobility_mcp/gtfs_shapes.py) 在 `route` 工具内 (`_leg_slices` 之后) 把每个 bus leg 的 chord wkt 换成真 GTFS shape 切段, orchestrator 透传/前端**零改动** (legs 形状不变只换 wkt 内容), 换过后 route 级 wkt/distance 从 legs 重导出。数据源 = 公开 tpl API (免认证, env `S4C_TPL_API_URL`)。**匹配全靠几何**: router 的 `trip_id`/`route_id` 属它自己的 feed 快照, tpl URI 属 km4city 自己的 ingest, 两边 id **不可对接** → 候选 = shortName 匹配 route_name (含 base-name 回退 "T1.3" vs "T1") 的全部变体, 接受条件 = 站点顶点到 shape 平均距离 ≤50m **且**上/下车有序投影 cost 小 —— **纯距离评分对同路床双向变体零区分力** (往返重叠), 方向只能靠有序投影暴露。切段保留**原边界顶点** (相邻 foot leg 共享顶点 + 车站 pin 挂点, 前端契约)。任何一环失败 → 该 leg 保持直线, 永不报错。缓存进程级, **conftest autouse 必清** (同 geocode LRU: 缓存命中会跳过 fake 队列)。"增强不上"别猜 —— 每站到候选 shape 的 min 距离一打印, 阈值误拒 (都略超) 和 feed 真漂移 (一站 1m 一站 755m) 一目了然。

L50 **多模式"选一条看详情"必须纯前端本地完成, 详情文本后端预渲染随 payload 下发** —— "点选 = 发一条 follow-up" 会重跑整轮 (含 bus 30-45s, 见 [[L31]]), 且 LLM 可能误抽 slot。修: 后端给 `data.routes[i].detail` 预渲染 (复用 `_format_detail`, 与单模式回复共用**同一渲染路径**不留第二处漂移); 前端 chips 点击本地 `drawRoutes([chosen])` + 弹 detail 气泡, "Mostra tutte" 恢复。两个前端陷阱: (a) chips 数据**必须 per-bubble closure 捕获** —— 用全局 `lastRoutes` + 事件委托的话, 旧气泡的 chip 会静默操作**新轮次**的数据 (标签与画出的线错位); 新轮次到达时旧气泡置 `.stale` 拒点。(b) 选择要**回写 history** (messages 是唯一跨轮状态, 不写则下一句 follow-up 看不到已选 mode), 但 inFlight 期间必须拒点 —— renderTurn 用响应整体覆盖 history 会丢掉飞行中 push 的条目。

L49 **GPS 就近排序救不回"结果集里根本没有"的候选** —— ServiceMap `/location/` 按纯文本相关度排序取 top-N, 一切 proximity 参数 (`position`/`selection`/`latitude`/`gps`) 实测无效, 锚点城市自己的同名街常**根本不在返回集里** (GPS 在 Firenze 时 "via Roma"→PRATO 12.8km, "via Mazzini"→MANCIANO 128km)。就近排序逻辑没错, 错在它只能在**给到手的集合**里选。修 (`_geocode`): 用户没点名城市且锚点城市可知 → **追加 ", {城市}" 重查一次** (把该城候选带进集合), 并用 `_signal_subset` 校验 —— 增强候选必须与原文共享 ≥1 个非路型词 token, 只共享 "piazza" 的属"城市匹配了但地点没匹配"→ 回退原结果, 保证 "Piazza dei Miracoli" 不被吸进 Firenze。锚点城市零额外成本: 起点缺省复用本就要调的 `coordinates_to_address` 的 `municipality`; 起点文本路径**惰性** reverse geocode 且**不进 audit**; 终点用已解析起点 feature 的 `city`。附带坑: mock 队列错位会被 `exec_tool` 的 catch-all 吞成静默假绿 → conftest 的 `make_client` teardown 断言队列耗尽。

L48 **"为未来准备的解析分支" = 从不执行的死代码, 且会伪装成数据通路骗过下游判定** —— 实例: `parse_service_features` 里那套"多空位 key + realtimeAttributes 双源扫描"是为"后端将来会在 search 结果里返空位"写的, 而 [[L33]] 的真机 probe **早已证明它永不返** (空位只在 `service_info_dev` 的时序 bindings 里), 该分支上线以来一次都没执行过 —— 却让 `free_spaces` 看起来"有来源", 使"这个字段有人填"的假设无人复核。同型: 假 bus arcs 骗过 `_pt_is_foot_only` (见 [[L39]])。教训: (a) 判断"这段会不会执行"要看**真机 probe 结论**而不是字段名的可能性 —— 写下"等后端支持了就能用"的分支, 实际是在赌一个已被证伪的假设; (b) 清理轮的产出不该只有"删了 N 行", 而应包含"删除过程中暴露的真 bug"。

L47 **心跳流式喂不活反代: jupyter-server-proxy 对非 SSE 响应整体 buffer** —— 读上游源码定案: `Accept == "text/event-stream"` 才走 `_proxy_progressive` (边收边转), **其余一律 `_proxy_buffered`** (收完整个 body 再一次性 write)。前端发的是 `Accept: application/json` → 心跳字节全攒在缓冲区, 上游反代看到的仍是整轮静默 → 60s 读超时切连接 (后端其实完全成功)。修: 改 **job + poll** —— `POST /advise` 只启动这轮并秒回 `{job_id}`, 前端每 1.5s `GET /advise/{job_id}` (202 = 还在算, 带 `stage`/`elapsed_s`; 200 = widget JSON)。**每个 HTTP 请求都亚秒 → 任何反代的任何缓冲/超时策略都掐不到**, 且不依赖 proxy 版本 (对"以后换成我们控制不了的一层"同样成立)。副作用: 长任务脱离请求生命周期 → `outputs.txt` 在 task 内写; job 表内存态 `JOB_TTL_S=300`。**禁改回单请求**。教训: (a) 保活方案要先证明"字节真的流出去了", 否则等于把死代码当修复交付; (b) 中间层行为是部署期变量, 稳的做法是把长任务从请求时长里**结构性移除**, 而不是想办法熬过每一层的超时。

L46 **Snap4City native 远程 `routing` 整体退役: foot/car/bus 统一走 mobility 的 What-If `route` 工具** —— 动机 = IO 统一 (两套响应形状/两套错误面) + km4city 稳定病灶。决策关键探针: 线上 whatif-router `foot` 0.46s / `car` 0.31s / `bus` 30.6s → 未 patch 的"每请求重建 PT 图"**只发生在 vehicle=bus** (foot/car 用 base 图), 比旧 native routing 还快 → 不必等 patch 即可切换。落地: `route(vehicle=foot|car|bus)` —— bus 保留 wkt 量距/instructions 时长/`_bus_arcs`/`_leg_slices`, foot/car 直取 `paths[0]` 真值 + 新 `_street_arcs` 出街道 arc (**漏这个则"主要街道"叙述静默断供**); mode 枚举统一 `foot`; km4city 的 retry/stale 梯与 hint 错误串匹配全删 (`_routing_hint` 只剩 `pt_degraded_to_foot`; ZTL 判断随之消失 —— GraphHopper 不认 ZTL, 无信号可译)。前端三 mode 全 manual 直画, widget 零 router 外呼。仍留远程: `coordinates_to_address` / `service_search_near_gps_position` / `service_info_dev`。教训: (a) "X 慢"要按**参数维度**归因, 3 发探针就把"要等 patch 才能切"证伪; (b) 双路由源 = 双错误面; (c) 换数据源前列一遍**派生数据消费者** (街道叙述/legs 画线/duration 排序), 每个都要有等价供给。

L45 **lesson 文件也会腐烂: 被代码推翻的 lesson 比没有 lesson 更危险** —— 审计实测: 多条讲的机制/文件已不存在, 另有几条被后续 lesson 直接推翻却没改。**危害不对称**: 缺 lesson 只让人重新踩坑, 错 lesson 会让人**绕开一条本来正确的路**, 而且 lesson 越权威腐烂越贵。三条规矩: (1) **改架构时同步改 lesson, 别只追加** —— 新条推翻老条时老条要**原地改写**, 而非留着让两条互相矛盾 (规则 7 的文档版); (2) **一条 lesson 只记一件事** (揉多件事进一行会制造断链); (3) **编号是外部 API** —— 源码注释、CLAUDE.md、README、前端 CSBL 都按 `L{n}` 引用, 所以过期条目**要么原地改写保号, 要么删干净后 grep 全仓修引用**, 绝不能重新编号。自查一行: `grep -o "\[\[L[0-9]*\]\]" docs/lessons.md | sort -u` 的每个编号都必须在 `^L{n}` 有定义。教训: 文档和死代码同理 —— **过期的那份不是"历史", 是噪音**, 历史归 git。

L44 **widgetMap `addCustomTrajectory` 有三条分支, 读上游源码一次破了两轮黑盒试错的误判** —— 三分支: (1) `'default'` = Leaflet Routing Machine 自路由, **写死 30s 超时** (线上 PT 请求 28-45s → 必超时, 这就是 bus 线不画的直接原因); (2) `mode.routing.manual: true` = 把给定点直接连线 (画现成几何的官方路径); (3) `drawMethod ≠ 'default'` = widget 自己 fetch router (无超时)。**终态 = 后端切好 walk/ride 几何, 前端走 manual 分支直画** (`_leg_slices` 按 PT instruction `interval` 索引 wkt 顶点切段 → `route["legs"]`), router 只被调一次、图与字同秒 (fetch 分支会让 router 被调两次, 已删, 见 [[L46]])。**[[L30]] 崩因更正**: 当年 `drawMethod:"custom"` 崩在部分点缺 `icon` 字段, 不是"多路点/drawMethod 不可行"。**附带数据事实**: 公交乘车段几何是**站到站直线** (一个顶点 = 一个站, 最长跳 476m, 同轮步行 leg 却紧贴路网) —— 不是切段 bug, 也不是 GTFS 缺数据 (两个 feed 都有 `shapes.txt` 且 100% trip 引用 shape_id), 而是 **GraphHopper 的 GTFS 导入器忽略 shapes.txt** (官方 PR #3127 至今未合, master 仍是站点连线)。服务端小修法: gtfs-lib 建图时已把 shapes 载入内存且有现成 `feed.getTripGeometry(trip_id)`, servlet 序列化 PT leg 处手里就有 `ptLeg.trip_id` → 挂个 `shape_wkt` ≈ 十几行 (给 referente 的 feature request)。客户端解已由 [[L51]] 落地。教训: 开源平台 widget 别黑盒试错, **拉上游源码读事件 handler**; "第三方返回的形状差"先开箱验数据 —— 数据在不在决定了要去催谁 (灌数据 vs 换引擎 vs 自己算)。

L43 **无城市无 GPS 时终点吃 server 排序会跨城误选 → 终点候选锚定已解析起点** —— "da via mortuli 40 a via pisana 166": origin 正确落 Firenze, 但同名街遍托斯卡纳, 无城市可收窄、无 GPS 可就近 → 吃 server 排序第一名 = **Lucca**, router 老实算出 65km 跨城公交 (路线对, 终点错)。修: `_geocode` 加 `anchor` 参数 —— **终点锚定已解析 origin 坐标** (一次行程两端通常相邻), origin 锚定 user GPS, 点名城市仍最优先。教训: "挑第一名"的 fallback 在候选跨城同名时是隐形炸弹 —— 挑选必须有**语义锚点** (另一端点/GPS/点名城市), server 相关性排序不是锚。

L42 **"数据上线" ≠ "patch 上线" —— 两件独立部署物, 切换默认端点前逐项实测** —— referente 报 GTFS 已灌后切回线上, 切换前公网实测 3 发: 真 PT leg ✅ 但**热请求仍 28-37s** → 只灌了数据没合 perf patch。而通用 `HTTP_TIMEOUT_S=40` 贴着峰值必然偶发超时 → `BUS_ROUTE_TIMEOUT_S=120` 单独兜住。教训: 对方说"部署好了"只保证他理解的那部分 —— 切生产依赖前用最小探针**分开验**返回形状与延迟, 且超时常量按实测尾延迟设, 别沿用为快端点调的值。

L41 **去 Toscana bbox 时 retry 必须一起删 (判据耦合); ServiceMap 的 proximity 参数对文本搜索排序零影响; near 工具无文本参数、类别名必须英文** —— 三个实证: (1) `geocode_with_retry` 的 transient 判据字符串是 "no Tuscany-area match", bbox 删掉后该 transient **无法定义** → retry 整块随之删除 (规则 7); (2) `/location/` 带不带 `selection`/`maxDists` 输出**逐字节相同** → GPS 最近的权威实现 = `_pick_coord` 客户端 haversine; (3) `service_search_near_gps_position` 是按 categories 过滤的 POI 搜索, **没有自由文本参数** (专名仍必须文本 geocode), 类别名必须英文 km4city 名 (`Pharmacy` 命中 / `Farmacia` 0 结果), 返回自带 `distance` 且已按距离升序 → 混合架构: 专名 geocode+haversine / 通用类别 near-tool 半径阶梯 (0.5→2→10km, 全空退文本 geocode)。**距离哨兵已删除**: 曾加 150km 上限防库外噪音, 但真机从 Brescia 查 "…a firenze" 时两端已正确命中却被 211km 整杀 —— **用户点名异地城市是合法查询**, 距离上限与"用户想查哪"语义冲突。整栈数据 (geocode 索引/路由图/GTFS) 都是 Toscana 范围 → 测试只用 Toscana 内地点。**两份索引别搞混**: referente 后端 = 联邦 SuperServiceMap (排序坏, [[L28]] 病根), 本地工具**有意**用单区健康的 `servicemap.disit.org/WebAppGrafo` (`S4C_SERVICEMAP_BASE` 可切)。附: 跨域 widget iframe 里 `navigator.geolocation` 需父 iframe `allow="geolocation"`; (0,0) 按无效 GPS 处理; 首轮有授权弹窗竞态 → 页面加载即预取一次定位。

L40 **whatif-router 的时刻字段是 UTC Instant 形 (带 Z), 叙述给用户前必须转 Europe/Rome** —— GTFS 班次语义是本地时刻, 原样引用差 1-2h (夏令时 +2/冬 +1)。修: `_rome_local_iso` 在 `_pt_stops` 解析时统一转带偏移 ISO (解析失败/naive 原样透传不瞎猜)。**orari programmati ≠ tempo reale**: whatif-router 只吃静态 GTFS (无 GTFS-RT 接线), referente 服务端 realtime 接口也空 (见 [[L33]]) → 回复里"informazioni in tempo reale non disponibili"是诚实措辞。端到端验证点: 首班上车时刻应 ≥ 查询时刻不久后, 差 ~2h 即时区错。

L39 **GTFS 已灌时短途 PT 也会返 walk-only (router 正确行为, 走路 Pareto 占优); 伪 bus fallback 删除, 退化如实呈现为步行** —— 1.3km 行程返纯步行 itinerary 不是 bug (GraphHopper PT 按最早到达优化, 短途走路完胜等车)。但 `_bus_arcs` 原有的 synthetic fallback 把步行街名**捏造成 `transport:"bus"` arcs** → 骗过 `_pt_is_foot_only` → `pt_degraded_to_foot` 不触发 → 把纯步行当"找到的公交路线"介绍。修: (a) 删 synthetic fallback, walk-only 如实返 foot arcs, `route["time"]` 放宽为 Σ>0 即给; (b) `_extract_data` 对 PT foot-only 从无条件丢弃改**重标 `mode:"foot"`**, 但仅当 `by_vehicle` 无真 foot (显式 bus 查询有绿线可画, 多模式查询真 foot 赢不重复); (c) `_route_pt` 的 `startdatetime` 钉 `ZoneInfo("Europe/Rome")` —— servlet 按本地时区解析无偏移字符串, 而 JupyterHub 进程是 UTC, 会差 2h 查错服务时段。教训: **"为兜底 X 而造的假数据"会在 X 消失后伪装成真数据骗过自己的判定链** —— 兜底宁可如实返退化形状让下游统一判定, 别在数据层伪装语义。

L37 **CSBL widget 的全局 CSS 选择器会泄漏进 dashboard 父页面** —— widgetExternalContent 是**直接贴进 dashboard DOM (非 iframe)** 的源码, 所以 `<style>` 里任何**非 `.adv`-scoped 的选择器**都作用于整个 dashboard 文档。回归实例: 加了一行 `html, body { height: 100%; margin: 0; }` 想让 `.adv` 撑满 → 覆盖了 dashboard 自身 `body` → **widget 标题栏不再渲染 + 父页面溢出产生幽灵滚动条**; 删掉该行同时修好两个症状。**规则 9**: CSBL `<style>` **只允许 `.adv`-前缀的 scoped 选择器**, 禁 `html`/`body`/裸标签/`*`; 需要撑满靠 host 给的容器高度。**另一条仍生效的分离**: 前端 `data` (widget 画图用) 与喂 LLM 的视图是**两条独立路径** (见 [[L12]]) —— 删前者不动后者。教训: 贴进共享 DOM 的 widget CSS 必须 scoped; 症状 (别的元素消失/多余滚动) 离改动点很远时, 靠 git `-S` 对症状字符串溯源最快。

L35 **bus 路线要全 multimodal 输出 (步行段 + 全站 + 真时长 + 门到门里程)** —— `_bus_arcs`: 连续非-pt 指令聚成一个 `foot` arc (`distance` 存**公里**, 与 route `distance_km` 同单位, `group_arc_legs` 累加才对), pt 指令出 board/alight arc (携 `line`/`headsign`/`stops`) → 序列 `[foot, bus, bus, foot]`。**时长 = Σ步行指令 `time` + Σ pt `leg.map.travelTime`**, 存 `route["time"]`, **不设 `eta`** (避 referente `eta`=到达钟点 的语义重载); GraphHopper `paths[0].time` **不用** (其 walking-pace 标定源自退化脚, 语义混)。**`distance` 改门到门 = wkt 折线 haversine 自算** (`_wkt_length_km`): `paths[0].distance` 只是**步行接驳段之和** (GTFS 不计车内里程), 直接用会低报整程。下游 `slim` 只留 board/alight + `stops_total`, 中间站全丢。教训: 距离/时长这种用户直读的数, **源字段语义要实测核对别照名取用**; 加特性时重验旧分支的触发条件是否仍成立。

L34 **whatif-router 的 `vehicle=bus` 退化成道路线 = 缺 GTFS 数据, 非请求参数错** —— 读服务端源码定位: `vehicle=bus` 本就走 GTFS 公交路由, 线上退化因实例没灌托斯卡纳 GTFS → 修法不是改我方路由, 是让 referente 灌数据 (已于 2026-07-10 完成)。**真 PT 响应结构 (实测, `_bus_arcs`/`_pt_stops` 按此解析)**: 公交段 = instruction `text:"Pt_start_trip"` 带 `leg.map:{type:"pt", route_name(线号), agency_name, trip_headsign, stop.myArrayList[].map.{stop_name, stop_arrivalTime}}`; 步行段 `leg:None` 只有 `street_name`。GTFS `stop_arrivalTime` 是时刻表**可靠值**。**GraphHopper 建图坑**: 首次 `/route` 请求**懒触发**建图 (deploy 完成 ≠ 图就绪), 且 SLF4J NOP logger 让建图进度全丢 —— 静默不是 hang。**限制**: 同运营商连续两程在 `group_arc_legs` 会并成一 leg (key=(transport,provider)), 换乘暂不拆。教训: 第三方路由"返退化结果"先分清是**请求参数错**还是**数据没灌** (读服务端源码定位触发条件 > 猜参数)。

L33 **car 路线附带"终点附近空停车位": 实时空位要二段取, 且 search 结果恒无空位** —— 仅 `"car" in modes` 触发。**并发坑 (关键)**: 必须 **flat** `asyncio.gather(*route_coros, parking_coro)` 把停车放最后一个, **不能嵌套** —— FakeClient 单队列 FIFO 且 `call_tool` 无 await 挂起点, flat 下协程按创建序跑完再下一个 (确定性); 嵌套则外层 gather 先调度 parking, 抢先 pop 队列 → 顺序乱、order 断言测试红。**真 envelope**: features 嵌在 `result[1].Services.features` (复数 Services!), `_find_feature_list` 递归找。**实时空位**: search 结果**不**带空位, 真值要调 `service_info_dev(serviceuri, fromtime="1-hour")` → `realtime.results.bindings`(**最新在前**)`[0].freeParkingLots.value` + `capacity.value`; 普通 POI 车库无 realtime → 留 None 退化显位置。enrich 后重排 (有空位者优先、多者前, 余按距离); enrich 调用**不进 audit**。距离客户端 haversine 从 dest 算 (envelope 距离字段单位不信)。类别 `Car_park`。教训: **"返空"别只测一个样本** —— 同类实体里 POI 无实时、IoT 实体有, 要按 URI 类型分别测; 实时当前值常在时序 bindings 里, 不在 attribute schema 里。

L32 **起终点 marker 飘逸的真凶 = 前端把 icon 钉在 routing-snapped 的 WKT 端点, 不是精确 geocode 点** —— 实测 geocode 本身已 civic 精确 (`/location/` 返 `serviceType:"StreetNumber"` 第一条), 是 routing 服务端把精确点 snap 到自家路网节点 (~27m 外), 前端又把 marker 钉在 `line[0]`/`line[last]`。修: 后端 execute 已算好精确 O/D, 经 `AdvisorState.endpoints` 透传, respond 注入 `data.origin`/`data.destination`, 前端 `buildTraj` 端点**优先用精确 O/D** (WKT 端点仅 fallback) → 多 mode 共享同一 O/D, 起终 icon 完美重叠。`data.origin/destination` 是给本项目自己前端画 marker 的字段 (消费者 = 我们的 dashboard HTML), 非对 referente widget 自创字段 (规则 8)。教训: "返空/不准"先看实测数据流定位真凶, 别凭印象改最上游 (geocode 没病, 病在前端取错端点)。

L31 **未指定出行方式 → 默认三模式全跑 (foot + car + public_transport), 墙钟 = 最慢那条** —— `mode = slots.get("mode") or "foot"` 会把"没指定"和"指定步行"压成一条 → 改用 `mode_specified = bool(slots.get("mode"))`。**默认包含 PT 是产品决定**: 三条线同图 + 三个时长对比是这个 advisor 的门面, 值得等; **代价明码标价** —— 线上 router 每 `vehicle=bus` 请求重建 PT 图 30-45s ([[L46]]), 缓解不是缩短而是**让等待可见** (202 带 `stage`/`elapsed_s`, 见 [[L47]])。显式 mode 仍只跑那一条 (问"a piedi"的人不该替 bus 的延迟买单)。**并发**: `_route` 只 return entry 不 side-effect append, 一把**扁平** gather (car 的停车搜索同 gather 且必须排最后, 见 [[L33]]), 结果按 modes 序 extend 保确定性; **`on_stage("routing_bus")` 必须在 gather 之前发一次** (塞进 `_route` 会被三个并发协程各发一次)。**假 PT 丢弃**: `_pt_is_foot_only` 判无非-foot transport leg → `_routing_hint` 给 `pt_degraded_to_foot`。**每个 waypoint 必须带 icon** (只给主线放会让非主线的点缺字段崩 widget, 见 [[L30]])。`routes` 按行程时间升序 (最快在前), 顶层 `wkt/mode/distance` = `routes[0]` 镜像 → 单路线消费者零改。教训: (a) "没指定" vs "指定默认值"是两种语义, 别在取默认那一步抹掉区分; (b) **"默认跑哪些 mode"是延迟预算 × 产品价值的联合问题** —— 按第三方修好后的稳态设计功能, 用 UX (进度可见) 兜住临时期, 并把代价写进文档而不是偷偷退化功能。

L30 **widgetMap 画线: 每个 waypoint 必须字段齐全 (尤其 `icon`); 崩因是缺字段, 不是"密集点/drawMethod 不可行"** —— 每点的 `mode.routing.graphhopper.type` 必须用前端 vehicle 名 `foot`/`car`/`bus` (填 routetype 原值或缺 `mode` → 读 `undefined.length` 崩); 每点都要带 `icon` (空串 = 无 marker)。当年"降采样 + `drawMethod:"custom"`"崩掉被误判成"这条路走不通", 实际那分支本身可行且更优, 崩只崩在部分点缺 `icon` (翻案见 [[L44]])。仍然有效: 喂 widget 不认的**顶层**字段照样崩 → 未验证字段先抓工作版 payload 或直接读上游源码, 别在真用户的图上试。教训: **"喂进去的形状不对"和"这条路走不通"是两个结论** —— 崩溃只证明前者, 别升级成后者写进 lesson, 否则后来者会绕开一条本来最优的路 (本条就是反例)。

L29 **forward geocode 走 mobility MCP server 包公开 km4city ServiceMap, 绕过 Snap4City native 坏索引 (server 现托管在 Snap4City server)** —— 裸 probe 证公开 ServiceMap (`servicemap.disit.org/WebAppGrafo/api/v1`, Snap4City 自家 What-If autocomplete 用的同一索引) 对地标/街道/车站稳中佛罗伦萨, 与坏 MCP 天差。修: [mcp_server.py](../src/snap4city_mobility_mcp/mcp_server.py) 用 httpx 包公开 ServiceMap, 返回**与 native 同形状** GeoJSON → 客户端解析全复用不改。**关键架构招 (避 [[L6]] 前缀税)**: 不把 mobility server 并进 native 的 client config (那会让 FastMCP 给**所有** tool 加前缀, 崩遍几十处裸名), 而是走**独立 single-server client** (`local_client` 注入参, 缺省回退) → 两 client 各单 server, 都保持裸名。两端点映射现有 2-pass: `excludePOI=True`→`/location/` (地址), `False`→`/api/v1?search=` 全文 (POI/地标)。**httpx 坑**: 全文 base (无尾斜杠) 返 **302**, 而 httpx **默认不跟随重定向** → `raise_for_status()` 抛错返空 (urllib 默认跟随, 故裸探针看不出) → `follow_redirects=True`。教训: 第三方 MCP 工具服务端坏且对方短期不修时, **自建 MCP server 包公开等价端点 + 保持同返回形状**是干净兜底 (下游零改), 且用独立 client 而非合并 config 能完全避开多 server 前缀迁移。

L28 **forward geocode (`address_search_location`) 是 referente 服务端 bug, 非 client** —— 裸 probe 对照同 query `via zara 3`: 公共端返 `VIA ZARA, FIRENZE` score 12.64 **第一**; referente MCP 返 **0 托斯卡纳** (top 是 Antwerpen/希腊 score 3-7) **且不按 score 排序**。schema 只有 `search/excludePOI/maxresults/logic/lang` —— **无 sort/order/bbox/region**, 且 `maxresults` 加到 5000 仍 0 命中 (托斯卡纳条目根本不在返回集) → 非 maxresults/POI/查询格式/lang 问题。**reverse `coordinates_to_address` 反而准** → 可靠路径是坐标→地址。**CORS 坑**: dashboard 在 `www.snap4city.org`, 桥地址带不带 `www` 算**不同源** → 跨域预检被 proxy 重定向到登录页 → `PreflightDisallowedRedirect` 崩; `BRIDGE_BASE` 必须和 dashboard **完全同源** (无预检、cookie 自动带)。教训: 接第三方 geocoder, 失败定位顺序 = 查询格式 → 参数 → maxresults → **裸 probe 对照同 query 在官方公共端 vs MCP 端**; 别假设 "MCP wrapper == 公共端", 排除全部 client 杠杆后才下"服务端 bug"定论。

L27 **dashboard 实时联动靠 `jupyter-server-proxy` 把 JupyterHub 内部端口暴露给浏览器** —— 浏览器进不来容器的 `localhost:8010`, 但容器**能出站**; cloudflared 没装、ssh 反向隧道要公网落点 → proxy 是唯一自助路。**老 jupyter_server (1.11) 上的安全配方** (上次装崩过服务器, 见 [[L15]]): (1) **版本钉 `>=3.2,<4`** (4.x 要 jupyter_server≥2); (2) **`pip install --no-deps`** 保证不升级现有 jupyter_server/jupyterlab/notebook (升级它们才是崩因); (3) 补装 `--no-deps` 漏掉的真依赖 `aiohttp`; (4) **预检不碰主服务器** —— 先 `jupyter server extension list | grep proxy` 应 `OK`, 再起一次性 `jupyter server --port=9999` 看扩展成功加载无 traceback, 确认后才重启主服务器; (5) 装/重启不会崩当前在跑的服务器 (扩展只在下次启动加载); (6) base 重启留存, 不用每会话重装。proxy URL = `{Jupyter base_url}/proxy/8010/...`。教训: 在老平台装 server 扩展, **`--no-deps` 锁死不升级 + 一次性服务器预检**两招把"崩了救不回"降到安全。

L24 **去 AI 感/简化注释时, LLM prompt 字符串是受保护的核心逻辑, 一字不能动** —— `UNDERSTAND_SYSTEM` / `_EXTRACT_SLOTS_SCHEMA` 的 description 读起来像"啰嗦的 AI 文本"(满是 em-dash、大写强调、否定式硬规则), 极易被当注释一起删, 但它们是**喂给 Llama4 的行为定义**, 措辞被反复调过, 改一个词就可能让分类/接地退化。规则: 去 AI 感只动**注释 / docstring / 文档散文 / 面向人的输出文本**; prompt 串、schema description、控制流、签名、阈值常量一律不碰。验证手段: `git diff` 里 grep prompt 内部短语必须 0 命中 = prompt byte-identical。教训: 看起来最像 AI slop 的文本 (长 prompt) 往往是调得最细的核心逻辑, 清理前先标出受保护区。

L23 **per-case 错误叙述逻辑别写进 prompt; 确定性判断下沉到 Python (altitude)** —— 曾让 LLM 对 `result["error"]` 串做模式匹配再决定建议, 这是典型的 "wrong altitude" (确定性逻辑硬编码进 prompt, 脆且靠 LLM 复述串)。`execute` 早已知 routetype, 且错误串确定性产自我们自己的工具层 → 判断移进 [`_routing_hint`](../src/snap4city_mobility_mcp/orchestrator.py): 按 (routetype, 串) 产 hint enum, 不匹配则不加。教训: 出现"当 error 串含 X 就说 Y"的措辞规则 = altitude 信号, 该逻辑属 Python。(respond 的 LLM 后已整体移除, hint 现由 `_compose_reply` 直接消费, 见 [[L57]]。)

L19 **"execute 对 mode 无关" ≠ "该 mode 实测能通"** —— plan 里凭"代码路径与 mode 无关"推断"happy path 已通", 但该 mode 从未实测成功过。定位"哪个 mode/参数真坏"最快的手段 = **绕业务层裸调工具看原始 payload** (绕开重试梯/slim/error 包装)。同源教训: (a) 用户报"慢/超时"先用带 elapsed 的裸 probe 证伪或证实, 别凭感觉调超时常量; (b) 显式传 schema 声明的默认值 = no-op 噪音; (c) **服务端能力会随时间漂移** (悄悄修好/悄悄加载), 结论要定期复测刷新别钉死旧口径。**LLM 接地**: 措辞节点 temperature 高 + "用户问 X 但数据里没有 X" 的缺口 = **编造温床** (曾凭空列出公交线号并杜撰运营商 "ATAF") → 低温 + **否定式硬规则** ("明说没有, 绝不补线号/运营商/站名" 比"别编"更可执行) + 语言规则提前到首句命令式。

L18 **LangGraph 节点返回的键必须在 State schema 里有通道, 否则被静默丢弃** —— `AdvisorState` (TypedDict) 没有的键, LangGraph 对 schema 外的节点输出**静默丢弃**, 下游永远收不到; 且**只有走 `graph.ainvoke` 全图才丢** (单测直接调节点函数不会暴露)。策略: 派生数据优先在消费端重推导 (单一来源现算), 要新增跨节点数据则**显式扩 `AdvisorState` 字段** —— 禁止"返回了就当传到了"。

L17 **km4city geocoder 把模糊 POI 排在真地点前, 跨城同名也会误中** —— `excludePOI=false` 让公司名排到真广场前 ("Piazza Duomo" 首条是一家公司), 地址 pass 又暴露跨城同名 (90km 外的同名广场)。四层修: (a) `_geocode_address_first` 先 `excludePOI=true` (纯地址条目, 坐标落在可路由街图上), 空才退 POI 兜底 (车站/地标只存在于 POI 目录); (b) `_pick_coord` 取 label 分词 (casefold + NFKD 去重音 + 去意语虚词) **⊆** search 分词的 feature, 带额外 token 的标签被拒, 无匹配才退 server 第一个; (c) `_narrow_by_city` 城市阶梯: 用户点名城市优先; (d) UNDERSTAND 把用户点名的城市留在 place text ("in Firenze" → "piazza Duomo, Firenze")。**geocode debug 日志必须带 picked feature 的 index + address** —— 吃过"日志只有坐标没地址"无法诊断的亏。

L16 **会话级对象别在 `run_advisor` 里每轮重建** —— 每轮都 `_build_config()` + `Llama4Client()` (= 新 TokenManager) 并重拉 `/apps.json`, 还每轮 `fetch_tool_schemas` (确定性图后已无人消费): token 本体早已持久化不会重登录, 浪费的是对象重建 + 网络往返 + stderr 噪音。修: orchestrator 模块级懒缓存 `_CFG`/`_LLM` (进程生命周期; token 过期刷新本就在 `TokenManager.get_token()` 内部按需做, 无 stale 风险); 删每轮 schema 拉取。MCP `Client` **刻意**仍每轮 `async with` 重连 (生命周期干净, 内网握手便宜)。诊断走 `logging.getLogger("snap4city_mobility_mcp")` → `api.py` 挂 FileHandler 写 `debug.log`, 不污染 widget JSON 形状。

L15 **JupyterHub 上 `pip install` 绝不进 base env** —— 一条命令能让整个 singleuser server 起不来。实录: 用 base 的 pip 装 `jupyter-server-proxy` (4.x 要 jupyter-server≥1.24) → 把 base 的 jupyter-server 从 1.11 强升到 2.18, 连带 jupyter-client/core/nbformat/traitlets 全升, 撞坏钉死老版本的 jupyterlab 3.1 / nbclassic / ipykernel。**当时不报致命错、server 还在跑** (改的是磁盘, 内存进程没动) → 一旦 Stop/Start 重新加载就崩 `Spawn failed: didn't respond in 30 seconds`。根因: 容器 overlay 在 stop/start 间**被复用** (非 fresh), 坏掉的 `/opt/conda` 持久 → 自愈无门, 只能让管理员重建容器。铁律: (a) 一切 `pip install` 只在专用 conda env (`s4c`) 内; (b) 装前看依赖约束, 会强升 jupyter-server 的别在老镜像上装; (c) 代码安全靠持久卷 `/home/jovyan`, 与 overlay 的 `/opt/conda` 分离。

L14 **多轮对话状态早已存在于对外输出的 `messages` (OpenAI 格式)** —— 加聊天前端时别"从头造多轮", 它只是把上轮 `messages` 回传当 `history` 的纯胶水。**同一时刻只留一个聊天入口** (加新入口时留着旧的 = 两份多轮实现, 违单一正确实现规则 7)。两个不变量延续到桥: (a) 回复文本取 `messages[-1].content` (OpenAI 标准), **不在前端硬编码** km·ETA 拼装; (b) 全量 widget JSON (含 `data.wkt`) 每轮写 `outputs.txt`、诊断写 `debug.log` 供离线看流程。教训: 选前端形态先问"运行环境能不能跑" —— 受限平台先用能跑通的过渡形态, 待底层路 (安全 proxy, 见 [[L27]]) 打通再升级到目标形态, 升级时按单一入口删掉过渡件。

L13 **Llama4 `tool_choice="auto"` 自由调工具不可靠** —— 模型一旦想叙述 (如"retrying with a different profile…") 就把 tool call 写成 **prosa 前缀 + pythonic 文本** 而非结构化 `tool_calls` → 漏成最终答案, `data` 空, 工具根本没跑; 打补丁解析只治标。**治本: 砍掉 `agent ⇄ tools` agentic 回路**, 改线性确定性图 `understand → execute → respond`: `understand` 用 forced `tool_choice={extract_slots}` (强制结构化, 从不走 pythonic) 只提取 slot; `execute` 纯 Python 按 mode 跑固定流 → LLM **永不自由调工具**, pythonic-leak 这条路彻底消失。

L12 **Llama4 后端 (vLLM) context 一大就炸, 且网关用 HTTP 200 包住上游 500** —— body 里是 `Error: 500 Server Error` 而 `_is_transient` 只认 502/503/504 → 不重试直接崩。根因常是请求太大: 工具原始返回**整个塞回** messages (geocode 100 个 feature / 超长 wkt), 每轮重读越堆越大。修: (a) `_TRANSIENT_HINTS` 加 `internal server error`/`server error`/`failed to make post request`; (b) [`slim_result_for_llm`](../src/snap4city_mobility_mcp/mcp_tools.py) 给**喂模型的 message** 瘦身 (geocode 留 top-5, routing 丢 wkt、arc 去重街道名), 而 **audit `tool_results` 留完整** → widget `data` 不受影响 (两路分离)。小 context = 少 500 + 少幻觉。

L10 **Llama4 旧 endpoint `llama4-inference` 即使 auth 成功 (200) 也返 `{"message":"Rule not found..."}` —— auth ≠ inference 授权** —— 用 OpenAI 兼容的 agentic endpoint `llama4-agentic-inference` ([llm.py](../src/snap4city_mobility_mcp/llm.py) 默认), 顺带解锁原生 tool calling。响应是 OpenAI `{choices:[{message:{content,tool_calls}}]}`。

L9 **JupyterHub 默认 kernel = Python 3.9, 太老装不了 fastmcp (要 ≥3.10)** —— `pip install fastmcp` 报误导性的 "no matching distribution"。建 conda 3.11 env + ipykernel (`s4c`)。git 可用, uv 通常没装 (用 pip/conda)。

L8 **(storico)** km4city routing 的"车在 ZTL 步行街"返裸顶层 `{"error": ""}` (wrapper bug, **稳定**不愈, 区别于冷启动 transient), 曾需要一整套重试/profile 互换梯来伺候 —— 该错误面已随 referente `routing` 退役整体删除 (见 [[L46]]), 保留编号供既有引用。

L6 **FastMCP 只在 multi-server cfg 才给 tool 名加 `<server>_` 前缀** —— single-server cfg 不加, 直接裸名 `call_tool("route", ...)` 即通; 一旦扩到 multi 会突然加前缀, 所有 hardcoded 名全崩 → `_build_config` 只挑 native 保持 single-server, 本地 server 另走独立 client (见 [[L29]])。

L5 **dashboard `/apps.json` 返回的 server URL 写死内网 IP** (`http://192.168.1.117:8000/...`), host 不同就连不上 —— [`_build_config`](../src/snap4city_mobility_mcp/mcp_tools.py) 拿到 cfg 后先把该 IP 替换成 `DASHBOARD_URL` 再喂 `Client`。

L3 **(storico)** km4city routing 冷启动/短时重复命中同坐标对时偶发返**临时**空 routes (几秒自愈), 曾用间隔重试梯遮掉, 与稳定失败的 [[L8]] 区分 —— 该重试梯已随 referente `routing` 退役删除 (见 [[L46]]), 保留编号供既有引用。
