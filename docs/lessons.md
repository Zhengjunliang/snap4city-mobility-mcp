L28 **dashboard 联动已端到端通; forward geocode (`address_search_location`) 是 referente 服务端 bug, 非 client** —— Step2 (NL 聊天框 + FastAPI 桥) 打通: 浏览器经 jupyter-server-proxy **同源**调 `api.py` 桥 (`https://www.snap4city.org/jupyterhub/user/<acct>/proxy/8010`)。**CORS 坑**: dashboard 在 `www.snap4city.org`, 桥地址带不带 `www` 算**不同源** → 跨域 OPTIONS 预检被 proxy 重定向到登录 → `PreflightDisallowedRedirect` 崩; 修=`BRIDGE_BASE` 必须和 dashboard **完全同源** (都 `www.`) → 无预检、cookie 自动带。桥每轮**覆盖** `outputs.txt`/`debug.log` (`api._reset_debug_log` 重建 handler + `_log_turn` 用 `write_text`) → 只留最后一条好查。**geocode 深查定论** (`scripts/probe_geocode.py`, 裸 probe 对照公共端 vs MCP): MCP `address_search_location` 与公共 `servicemap.disit.org/WebAppGrafo/api/v1/location/` **结果完全不同** —— 同 query `via zara 3`: 公共端返 `VIA ZARA, FIRENZE` score 12.64 **第一**; MCP 返 **0 托斯卡纳**, top 是 Antwerpen/希腊 score 3-7, **且不按 score 排序** (Santa Croce 的托斯卡纳 score 12.19 被埋在 `@#2714`, 排在法国 score 7.16 之后)。schema 参数只有 `search/excludePOI/maxresults/logic/lang` —— **无 sort/order/bbox/region** → client 无法让服务端排序或限区; `maxresults` 加到 5000 对稀有名 (`via zara`/`SMN`) 仍 0 托斯卡纳 (Tuscany 命中根本不在返回集) → **非 maxresults 问题**, 也非 POI/查询格式/lang/logic (全试过)。结论: forward 名字→坐标对**有辨识度地名** (Duomo/Carducci) 能中, 对**常见/稀有歧义名** (Santa Croce/via zara/SMN) 服务端失败 → 报 referente (请按 score 排序 / 限 km4city 托斯卡纳索引 / 加 bbox 参), client 无解。**reverse `coordinates_to_address` 反而准** (点 → `{"result":[{address:"VIA ZARA", municipality:"FIRENZE",...}]}`) → 已入 `EXPOSED_TOOLS` + `mcp_tools.reverse_geocode(client, lat, lng)` helper; 可靠路径是**坐标→地址** (未来 GPS/点地图 near-me), 不是 forward。本轮 geocode 调优 (`GEOCODE_LANG="it"`/`GEOCODE_LOGIC` 常量加进两 pass) 仅对能中的 query 有微益, 非解, 保留无害。教训: 接第三方 geocoder, 失败定位顺序 = 查询格式 → 参数 → maxresults → **裸 probe 对照同 query 在官方公共端 vs MCP 端**; 别假设 "MCP wrapper == 公共端", 排除全部 client 杠杆后才下"服务端 bug"定论。考试 demo 挑能跑通的辨识度地名即可。

L27 **dashboard 实时联动靠 jupyter-server-proxy 把 JupyterHub 内部端口暴露给浏览器; 老 jupyter_server 上装它的安全配方** —— 浏览器(dashboard 在 snap4city.org)**进不来** JupyterHub 容器的 `localhost:8010`(桥), 但容器**能出站**(git/pip/LLM 都是出站)。开发期自助开入站口 = `jupyter-server-proxy`(终态仍是 referente 同源托管, 但每次迭代不能等 referente)。cloudflared 没装、ssh 反向隧道要公网落点+挂公网不安全 → proxy 是唯一自助路。**坑+安全配方**(此机 base=jupyter_server **1.11.0** 很老, 上次装崩过服务器 L15): (1) **版本钉 `<4`** —— 4.x 要 jupyter_server≥2, 装 `jupyter-server-proxy>=3.2,<4`(实得 3.2.4)。(2) **`pip install --no-deps`** 装 proxy+simpervisor → **保证不升级现有 jupyter_server/jupyterlab/notebook**(升级它们才是崩因); 老 pip 无 `--dry-run` 故靠 --no-deps 兜底。(3) `--no-deps` 漏的真依赖 **`aiohttp`** 要补装(`pip install aiohttp`, 它只拉 async 叶子库不碰 jupyter)。(4) **预检不碰主服务器**: 装进 base 后先 `jupyter server extension list|grep proxy`(应 `OK`) + 起一次性服务器 `jupyter server --port=9999 --no-browser --ServerApp.token=''` 看 `jupyter_server_proxy | extension was successfully loaded` 无 traceback → 证明主服务器重启会干净; 主服务器只在确认后才 Hub Control Panel→Stop/Start 重启激活。(5) 装/重启**不会**崩当前在跑的主服务器(扩展只在下次启动加载); 关 terminal 也不崩(terminal≠server)。(6) 实测 **base(`/opt/conda`) 重启留存**(s4c env + proxy 重启后都在) → 不用每会话重装。proxy URL = `{Jupyter base_url}/proxy/8010/...`(如 `…/user/mobilityadvisor/proxy/8010/health`)。撤销 `pip uninstall -y jupyter-server-proxy simpervisor aiohttp`。教训: 在老平台装 server 扩展, **--no-deps 锁死不升级 + 一次性服务器预检** 两招把"崩了救不回"(referente 不帮、重启不还原的假设)降到安全。

L24 **去 AI 感/简化注释时, LLM prompt 字符串是受保护的核心逻辑, 一字不能动** —— 给老师对接前做了一轮去 AI 感 (砍 essay docstring、删内联 lesson 标签、收 em-dash、去 emoji `📍/⚠`)。陷阱: `UNDERSTAND_SYSTEM`/`RESPOND_SYSTEM`/`_EXTRACT_SLOTS_SCHEMA` 的 description 读起来像"啰嗦的 AI 文本"(满是 em-dash、大写强调、否定式硬规则), 极易被当注释一起删, 但它们是**喂给 Llama4 的行为定义**, 措辞被 L19/L23 反复调过, 改一个词就可能让分类/接地退化。规则: 去 AI 感只动**注释 / docstring / 文档散文 / 面向人的输出文本**; prompt 串、schema description、控制流、签名、阈值常量 (`temperature`/`*_RETRIES`/`TUSCANY_BBOX`) 一律不碰。验证手段: `git diff` 里 grep prompt 内部短语 (如 `Classify by STRUCTURE`/`Ignore greetings`) 必须 0 命中 = prompt byte-identical。emoji 去除仅限**无 test 断言**的 fallback 输出 (`_template_answer`), CLI 标记 (`✦`/`✗`) 属正常终端 UX 保留。另: `lessons.md` 本身和 walkthrough 末尾的 lesson→code 索引表是合法方法论归档, 不算"内联 AI 标签", 保留; 只删散落在代码注释里的 `(L19)` 式小标。教训: 看起来最像 AI slop 的文本 (长 prompt) 往往是调得最细的核心逻辑, 清理前先标出受保护区。

L23 **per-case 错误叙述逻辑别写进 respond prompt; 确定性判断下沉到 `_results_view` 的 `hint` (altitude)** —— RESPOND_SYSTEM 曾让 LLM 对 `result["error"]` 串做模式匹配再决定建议 (`"no route found (empty routes list)"`+car → ZTL 建议步行/公交 vs `"empty response from routing service"` → 服务端问题禁断言 ZTL), 这是 Anthropic 说的 "wrong altitude" (确定性逻辑硬编码进 prompt, 脆且靠 LLM 复述串)。`execute` 早已知 routetype + 串确定性产自 [mcp_tools.routing](../src/snap4city_mobility_mcp/mcp_tools.py) → 把判断移进 [_routing_hint/_results_view](../src/snap4city_mobility_mcp/orchestrator.py): error 分支按 (routetype, 串) 加 `hint` enum (`car_pt_blocked_try_foot` / `service_empty_try_foot_or_later`), 不匹配则不加 (respond 退通用错误规则)。prompt 只剩一句"按 hint 建议; 无 hint 别臆断 ZTL"。`hint` 仅在内部 RESULTS view (传 respond LLM 的中间数据), `response.data` 走独立 `_extract_data` → 不进对外 widget, 不违规则 #8。L19 不变量 (car/PT 空 ≠ ZTL) 守护点从 prompt-串断言移到 `_results_view` 行为断言 (`test_results_view_hint_separates_service_error_from_ztl`, 更强)。配套 prompt 精简: UNDERSTAND/RESPOND 去 prompt⇄schema 冗余 (分类语义只留 schema enum description, 见 memory feedback_prompt_lean_refactor), 内联示例提到 `<examples>` few-shot 块。教训: prompt 里出现"当 error 串含 X 就说 Y"= altitude 信号, 该逻辑属 Python; prompt 只表达"读结构化字段照做", 不替代码做串匹配。
