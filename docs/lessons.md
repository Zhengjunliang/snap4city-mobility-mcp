# Lessons — 架构陷阱沉淀

---

L2 km4city OpenAPI spec (`ascapi-openapiv3.json`) 跟真实 `/shortestpath` 响应**严重偏差** — spec 说 `journey.routes` 是 dict 含单字段 `wkt`, 实际是 list of dict, 每个 dict 有 `arc/distance/eta/time/wkt` 5 字段; 顶层 spec 列 2 字段 (`pathsearch_time`/`journey`), 实际 8 字段 (多出 `elapsed_ms`/`elapsed_osmdst_ms`/`elapsed_osmsrc_ms`/`node_id_time`/`message_version`/`response`)。另外 `ok=True` 不等于 "找到路" — `routes` 可以是空 list (车在步行街 / 起终点同一点都会触发, API 不抛 4xx)。根因: spec 由人手写, 真实部署版本更新但 spec 没跟。修复: (a) 任何 km4city endpoint 在写工具前**必须 live probe** 一次再定字段, 不能只信 spec; (b) 上层 caller (M3 orchestrator) 必检 `len(payload["journey"]["routes"]) > 0` 而不是只看 `ok`; (c) 工具层不做字段裁剪 (referente 风格) 反而避免了 spec 偏差伤害 — 把整 dict 透传给上层, 上层按真实结构挑。代价 ≈ M2.0 多花 1 步 live inspect (5 min), 收益是 M3 orchestrator 不会因 `routes[0]` 解 dict 失败崩。

L3 M3 验证矩阵跑 V2-V6 后再跑 V2 happy walk 偶发返 "no route found (empty routes list)", 但**直接 stdio Client probe 同输入同时间立刻 0.796 km 路径成功** — 根因疑似 km4city `/shortestpath` 在短时间内连续命中相同 endpoint (V3 用 src==dst 同坐标, V4 用 Piazza Duomo 同 origin) 后**对相同坐标对返临时空 routes**, 类似一种隐式 short-window 缓存或速率限制旁路, 几秒后自愈。**修复 / 工作流**: (a) 端到端验证矩阵不要把 V2 happy 放在一堆 src==dst / 同坐标边界用例之后立刻跑, 应**先 happy 再边界**, 或边界批跑完隔 ≥5s 再回归 happy; (b) 看到 happy walk 单次空 routes 时**先用 stdio probe (即 `_probe_m3.py` 那种 ~10 行直调) 复现一次**确认是真 bug 还是 API 状态污染, 别立刻回滚代码 — 我差点为此回滚一次 _resolve_endpoint 重构。代价 ≈ 0 (工作流改动), 收益是避免无谓回滚。**M4 复发**: 改完 `pyproject.toml [project.scripts]` 加第二条 script entry 后, 跑 M4 V1 (happy walk) **首次冷启动**也命中空 routes — 间隔 8s 重跑立刻 0.68 km 成功, 跟 M3 现象完全一致。这说明 short-window stale 状态**跟前序验证用例污染无关**, 是 km4city 端的天气, 任何**冷启动 cli 首发命中相同坐标对**都有概率触发。reproducer 进一步收紧。

L4 `[project.scripts]` 新增 / 改条目后, `.venv\Scripts\<name>.exe` 启动器**不会自动**因源码改动重新生成 — 启动器是 `uv pip install` 阶段从 `[project.scripts]` 写出的二进制 stub, 与 editable `pth` 文件指 src/ 是两件独立事情。直接跑 `<name>` (启动器路径) 会报 `'<name>' is not recognized`, 容易误诊为 PATH 问题。**修复路径** (任选其一): (a) 显式 `uv pip install -e .` 重装一次, 重新生成启动器; (b) **`uv run <name>` 即可** — 实测 uv (0.11.x+) 在检测到 `pyproject.toml` 比 lock 状态新时会自动 sync + 重建 editable wheel + 注册新启动器, 整个过程对调用者透明 (M4 验证 V6 直接 `uv run snap4city-mobility-cli` 一次成功, 没单独跑 `uv pip install -e .`)。**注意分清**: L4 跟 README §9 "AttributeError: module ... has no attribute 'main'" 那条**不同根** — 后者是 entry-point 字符串拼写错 (`<pkg>:main` vs `<pkg>.<mod>:main`), L4 是"字符串对但 stub 没生成"。代价 ≈ 一次性 `uv run` 触发或 `uv pip install -e .` 显式, 收益是避免下次有人加第三条 script 时再踩同一坑。

L5 (Phase 5 §2 R0 探针 2026-05-28) FastMCP dashboard 自带 `/apps.json` 端点返回的 server URL **写死内网 IP** (这里是 `http://192.168.1.117:8000/...`), 而不是 dashboard 自己暴露给客户端的 host。在 SSH tunnel 拓扑下 (`ssh -L 8000:192.168.1.117:8000 ...`) 客户端只能走 `localhost:8000`, 把 `apps.json` 原 URL 直接喂 `Client(cfg)` 必出 `Client failed to connect: All connection attempts failed` + `list_tools` 返 0 工具。**根因**: dashboard Python Example 文档假设客户端跑在 VPN 内网, 不是 tunnel 端。**修复**: 拿到 cfg 后第一件事就是把所有 server `url` 里的 `http://192.168.1.117:8000` 替换成 `http://localhost:8000` 再喂 Client。[orchestrator._build_config](../src/snap4city_mobility_mcp/orchestrator.py) 已封装。代价 ≈ 一行 `str.replace`, 收益是避免下次接其他 referente 部署 (或同部署换 host) 时再踩 20 分钟"连不上"。

L6 (同 R0 探针) FastMCP `Client(cfg)` 的**工具名前缀策略**是 dashboard Python Example 一行警告 (`# "tool_A" becomes "Project_A_tool_A"`) 没说清楚的: **只在 multi-server cfg 时**才加前缀 (避免 tool 名碰撞), single-server cfg **不加**。我们 cfg 只塞 `snap4agentic_advisor_native` → 直接 `call_tool("address_search_location", ...)` / `call_tool("routing", ...)` 不带前缀就通。若未来 cfg 扩到 multi (native + experimental + legacy), prefix 会**突然出现**, 所有 hardcoded 工具名调用瞬间崩。**修复路径**: (a) 写 orchestrator 时假设 single-server, 把 cfg 收窄做法显式封装 (我们 `_build_config` 就是这么做的, 只挑 native); (b) 若要 multi, 引入工具名常量 + cfg 启动期一次性 `list_tools()` 检测前缀模式, 拼实际名。代价 ≈ 写 `_build_config` 时多一句过滤, 收益是避免日后扩展时所有 call_tool 一片 KeyError。

L7 (Phase 5 §2 R4 retest 2026-05-28) referente `routing` 工具的返回 envelope `response.error_message` 在**成功路径下也是非空字串** `"successful"`, 而 `response.error_code` 是 `"0"`。本能写 `if err_msg or err_code != "0": treat as failure` 就 100% 把成功误判成失败 — 我 R4 第一发就踩, ` ok=false "routing failed: successful (code=0)"` 自相矛盾输出。**根因**: km4city 的 envelope 设计是状态字段二选一不重叠 — `error_code` 是唯一权威 (`"0"` = OK, 任何非零 = 失败), `error_message` 永远填 (成功时填 "successful"/"OK"/类似, 失败时填具体错). **修复**: failure 判断**只看 error_code**, `err_msg` 仅做错误时的展示文案。代价 ≈ 删 `if err_msg or` 一个条件, 收益是 happy 路径不再被自家代码秒杀。注意这跟 L2 (OpenAPI spec 偏差) 同根 — 都是 km4city/referente 字段语义不直观, 必须实测确认而不能凭名字猜。

L8 (Phase 5 §2 R4 retest 2026-05-28) referente `routing` 工具对**车在中心步行街** (Florence ZTL Duomo→Santa Croce car) 的响应**不是** km4city 风格的 `{journey: {routes: []}, response: {error_code: "-2"}}` (旧 stand-in 在同输入下走 km4city 直 API 时返这个), 而是**裸 `{"error": ""}` 顶层空 body** — 这是 referente wrapper 在 km4city car routing 内部失败时**封装路径有 bug**, 没把 km4city 的 -2 envelope 透传出来, 直接吃掉返空字符串错误。我们 R4 实测加 1 次 6s 重试也不会自愈 (跟 L3 transient stale 不同, 这个是稳定 bug), 我们这边表现成 `routing failed: empty body (L3 stale didn't clear after retry)` — 错误文案误导, 实际不是 stale。**修复路径 (我们这边)**: 暂时无 — referente wrapper 不开源, 等他们修。**绕过**: (a) 应用层显式标 known issue, 文档 README §9 告知用户 car-ZTL 期望失败但错误文案有误; (b) 若 referente 后续修了, 我们 Shape A 自动会捕到正常 envelope 不用改。**注意分清**: L8 跟 L3 同样产生 "empty body" 表象, 但 L3 是 transient (重试自愈), L8 是 stable (重试也不愈, referente wrapper bug); 区分手段: 同一调用第三次仍空 body → L8, 隔 5-10s 第二次就有 envelope → L3。

L9 (2026-06-03 接入 JupyterHub runtime) Snap4City JupyterHub 默认 kernel = **Python 3.9.7**, 装不了 fastmcp — `pip install fastmcp` 报 `ERROR: No matching distribution found for fastmcp (from versions: none)`。**根因**: fastmcp 2.x/3.x `requires-python >=3.10`, 3.9 下所有发行版被 pip 按 requires-python 全过滤, 报错文案像"包不存在 / 没网络"实为版本不兼容, 极易误诊成 PyPI 不通。**修复**: JupyterHub 里用 conda 建新环境 `conda create -y -n s4c python=3.11` → `conda run -n s4c pip install fastmcp langgraph httpx requests ipykernel` → `conda run -n s4c python -m ipykernel install --user --name s4c` → notebook `Kernel → Change kernel → Python 3.11 (s4c)`。代价 ≈ 一次性建环境 (~1-2 min), 收益是别在"PyPI 不通"上瞎查半天。注意 git (2.25.1) 在 JupyterHub 可用, uv 大概率没装 → 用 pip/conda。

L10 (2026-06-03 Llama4 接入) LLM 旧 endpoint `llama4-inference` 被弃用: auth (OAuth2 password grant) 返 **200 成功**, 但 inference POST **极快** (~0.1-0.3s) 返 `{"message": "Rule not found for this user/path"}`。**根因**: referente 把规则迁到新的 OpenAI 兼容 endpoint `llama4-agentic-inference`, 旧 path 的规则没绑功能账号。**易误判**: auth 成功 ≠ inference 授权, 两段独立鉴权, 别因为"拿到 token 了"就以为通了; 0.x 秒极快返回 = 网关层规则拒绝 (没进模型)。**修复**: 用 agentic endpoint (`clearml_ondemand_api_base_url=https://www.snap4city.org/apis/llama4-agentic-inference` + `clearml_llm_endpoint=llama4-agentic-inference`; [llm.py](../src/snap4city_mobility_mcp/llm.py) 默认已切)。响应也从 legacy `{answer}` 变 OpenAI `{choices:[{message:{content,tool_calls}}]}`。代价 ≈ 改 2 个 config 字段, 收益是顺带解锁原生 tool calling (LLM 自己选 MCP 工具)。
