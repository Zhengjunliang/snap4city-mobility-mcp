# Lessons — 架构陷阱沉淀

> 每条 `L{n} {仍生效的绑定事实 + 修复}`. 只留仍约束当前设计的坑。

---

L22 **foot 路由失败别再怀疑 geocode; foot 覆盖是"中心城区绑定"非纯距离阈值** —— chat 里 SMN→Rifredi foot 全空, 曾疑 "stazione di Rifredi" 被 geocode 成街道 "VIA DI RIFREDI" 坐标落步行图外。裸探针 (`scripts/probe_routing.py`) 对照**真实火车站坐标** (43.79556/11.2436) vs **geocode 街道坐标** (43.795155/11.240046) → 两者 foot 全返 `{"error":""}` → **geocode 没错**, 是服务端 foot 路由覆盖不到 Rifredi (中心北 ~2.5km)。即 foot 不只 L19 的远程 (10km) 失败, **中等距离的非中心区也失败** → 覆盖按"中心城区"而非硬距离阈值; 实测通的全是中心 1-2km (Duomo/SantaCroce/CampoMarte/SMN)。修: **无 client 可修, geocode 不动** (撤销原"另开任务修 geocode"猜想), respond 已诚实报 empty-routing (L19)。附: 本轮 probe 的 TOOLS 列表显示服务端**新增工具** `transport_routes_search_near_gps_position`/`_within_gps_area`/`_within_wkt_area` + `tpl_routes_by_stop` + `service_search_near_gps_position`/`coordinates_to_address`/`distance_from_coordinates` 等 → 之前"按地点/坐标查附近站/线"的能力缺口现有工具可接 (另开 feature 任务)。教训: 诊断 routing 失败先用裸探针对照"已知好坐标"再定 geocode vs server, 别预设 geocode; 且失败前先看 TOOLS 是否已扩 (服务端会悄悄加工具)。

L21 **tpl 链解析: 别信 tool doc 的示例 URI, 先裸探真表; 名字匹配用双向 token** —— `tpl_lines` 文档里的示例 agency `.../km4city/resource/AtF` 线上**不存在**; 品牌 "Autolinee Toscane" 拆成 ~40 子网 (`ExtraUrbano <省>` / `Urbano <市>` / `Linee Regionali`), Firenze 城区 = `…_Agency_888-48` (实证: line "6" + 888-48 → 22 routes; Arezzo → [])。旧 `_resolve_agency` 命名匹配方向反 (品牌名是用户输入的子集、子网名是超集) → 永不匹配 → LLM 拿 agency 列表编造假线路。修 ([tpl._resolve_agency](../src/snap4city_mobility_mcp/tpl.py)): 双向 token 匹配 (`toks<=want or want<=toks`) + 多匹配优先 `_is_florence_urban` + 空文本默认 Firenze 城区。实证真字段: `tpl_routes_by_line` item 键是 `route`/`wktGeometry` (非 `wkt`); `tpl_stops_by_route` GeoJSON 嵌在 `BusStops` 键下、无顶层 `type`, 名/URI/坐标在 `properties` 里 (`_features` 容错下钻); `tpl_stop_timeline` 给 `{stop, lines}` 但 `timetable`/`realtime` 线上**空** (schema 无 datetime 参数, 真实发车时刻疑似服务端未加载, 待 referente)。`_match_stop`: 站名通常比用户输入**长** ("San Marco" → 官方 "Museo Di San Marco"), 故 `用户词 ⊆ 站名` 优先 (与 geocode `_pick_coord` 的 `label ⊆ search` 相反)。客户端侧测试全闭环; 剩 car/PT routing 空 (L19) + timetable 空两项纯服务端, 交 referente。教训: 解析"名字→实体"前先裸探针 dump 全表看真实命名/分网, 别信 tool doc 的示例 URI, 且 dump 要够长 (截断会把关键第二段切掉, 伪造"无数据"假象)。

L20 **referente geocoder 时间维度非确定性 (同串时好时坏); 别从单次失败诊断"查询形状"问题; 真修是 flaky 重试** —— "Università di Firenze, Viale Morgagni, Firenze" 某次 geocode 返 100 条全希腊/西班牙、零托斯卡纳 → 当时误判成"机构长名固定难定位", 加了 UNDERSTAND_SYSTEM "机构名转街道" 规则。裸探针 (绕 orchestrator 直接 2-pass) 几分钟后复测同一 exact 串 → raw 第一条就是正确的 VIALE MORGAGNI FIRENZE; 即**后端对同一查询的返回随时间漂移**, 非机构 vs 街道的系统差异。修: (a) **撤掉"机构名转街道"误诊规则** (街道名 raw 也 flaky, 转写救不了, 徒增 prompt 复杂度 — 违单一正确实现); (b) client 2-pass+bbox+城市阶梯 (`_geocode_address_first`) 已能在"后端返任意一条托斯卡纳"时捞回, 唯一真失败 = 后端返 **0 条托斯卡纳** 的 transient 窗口 → 加 `geocode_with_retry` (仅当结果含 `"no Tuscany-area match"` 时有界重试, `GEOCODE_FLAKY_RETRIES=2`)。教训: 诊断"查询形状坏"前先用裸探针多次复测同一输入 — 后端非确定性会把 transient 伪装成确定性 bug。

L19 **"execute 对 mode 无关"≠"该 mode 实测能通"; 哪个 mode 真坏靠裸 `call_tool` 探针 (绕 orchestrator) 定位** —— car 分流程 plan 凭"`execute` 代码对 mode 无关"推断"car happy path 已通", 但项目记录里 car **从未实测成功出过路线**。裸探针 (绕 orchestrator 的重试梯子/slim/error 包装, 直接 `call_tool("routing", {...routetype})`) 跑 OD × mode 矩阵 → **car 和 public_transport 对任意 OD (含非-ZTL 的 Campo di Marte) 全返裸 `{"error":""}`, 而同一 OD 的 foot_quiet 正常出 journey**。即: 后端活着, 是 car/PT 这两个 mode 服务端 wrapper 专门坏, 非 ZTL/非坐标/非全域宕机, client 调用 (工具名/参数/routetype 合法值) 全对 — 服务端问题, 上报 referente。**2026-06-12 决定性复测 (5 OD × 4 mode, 含纯郊区 Sesto→Scandicci、城际 Peretola→Prato、短途 CampoMarte→Coverciano)**: car 5/5 全空, public_transport 5/5 全空且加 `startdatetime` 仍空 → car/PT 服务端死板上钉钉, 非 ZTL/距离/坐标。foot 城区短中程正常; 远程 (10-12km) 失败属**设计内** (后端 `journey.search_max_feet_km` 极小, 步行路由本就只搜短程)。教训: (a) plan 里"已通"必须有实测支撑, "代码路径无关"不是实测; (b) 定位"哪个 mode/参数真坏"最快手段 = 绕业务层裸调工具看原始 payload; (c) respond 措辞按 `route_error` 串区分: `"empty response from routing service"` (L8 服务端空) 禁断言 ZTL, 只 `"no route found (empty routes list)"` (L2 真空 routes) 才说步行区 — [RESPOND_SYSTEM](../src/snap4city_mobility_mcp/orchestrator.py) 已拆分。

L18 **LangGraph 节点返回的键必须在 State schema 里有通道, 否则被静默丢弃** —— 设计 tpl missing-slot 路径时差点踩中: `execute` 想返回 `{"missing": [...]}` 给 `respond`, 但 `AdvisorState` (TypedDict) 没有 `missing` 通道 → LangGraph 对 schema 外的节点输出键**静默丢弃** (版本不同也可能直接报错), `respond` 永远收不到, 且测试 mock 单测节点函数时不会暴露 (只有走 `graph.ainvoke` 全图才丢)。修复策略: **派生数据不进 state** — `respond` 端从已有通道重推导 (missing slots 由 `_missing_slots(intent, slots)` 从 `slots` 通道现算, 必需 slot 表 `tpl.REQUIRED_SLOTS` 单一来源供 `run_tpl_flow` 跳链和 respond 提问共用)。要新增跨节点数据, 要么进已有通道 (如塞进 `slots`/`tool_results`), 要么显式扩 `AdvisorState` 字段 — 禁止"返回了就当传到了"。

L17 **km4city geocoder 把模糊 POI 排在真地点前, 跨城同名也会误中 — geocode 两段式 + 选点按 label 匹配 + 城市阶梯** —— `excludePOI=false` (L11 为找地标所设) 让 POI 排到真广场前 ("Piazza Duomo" 首条是公司 `PIAZZA DUOMO DI PRIZIO STEFANO...`, 偏西 1.1km → 步行 1.83km vs 0.68km 基准); 地址 pass 又暴露**跨城同名** ("Piazza Duomo" 首位是 90km 外 CASTELNUOVO/PIETRASANTA 的同名广场, "via dello Steccuto" 返 100 条全荷兰 Maastricht)。修四层: (a) [mcp_tools._geocode_address_first](../src/snap4city_mobility_mcp/mcp_tools.py): 先 `excludePOI=true` (纯地址条目, 坐标在可路由街图上), Tuscany 区内空才退 `excludePOI=false` 兜底 (车站/地标只存在于 POI 目录); (b) [orchestrator._pick_coord](../src/snap4city_mobility_mcp/orchestrator.py): 取第一个 address/name 分词 (casefold + NFKD 去重音 + 去意语虚词) **⊆** search 分词的 feature — 带额外 token 的标签被拒, 无匹配退回 server 第一个; (c) [mcp_tools._narrow_by_city](../src/snap4city_mobility_mcp/mcp_tools.py) 城市阶梯: 用户点名城市 (city 分词 ⊆ search 分词) > FIRENZE 默认 > 全托斯卡纳; (d) UNDERSTAND_SYSTEM 把用户点名的城市留在 place text ("in Firenze" → "piazza Duomo, Firenze")。另: geocode debug 日志必须带 picked feature 的 index+address — 诊断吃过日志只有坐标没 address 的亏。

L16 **会话级对象别在 `run_advisor` 里每轮重建** —— chat.py 实测每个回合都打一遍 `[INIT]/[LOAD_TOKEN]` 噪音并重拉 `/apps.json`: 根因是 `run_advisor` 每次调用都 `_build_config()` + `Llama4Client()` (= 新 TokenManager), 还顺带每轮 `fetch_tool_schemas` (确定性图后 schema 已无人消费, 纯多一个 round trip)。token 本体早已持久化在 `token_stored.json` (不会重登录), 浪费的是对象重建 + 网络往返 + stderr 噪音。修复: orchestrator 模块级懒缓存 `_CFG`/`_LLM` (进程生命周期, token 过期刷新本就在 `TokenManager.get_token()` 内部按需做, 无 stale 风险); 删每轮 schema 拉取; TokenManager 命中有效 token 的热路径静默。MCP `Client` 刻意仍每轮 `async with` 重连 (生命周期干净, 内网握手便宜)。诊断走 `logging.getLogger("snap4city_mobility_mcp")` → chat.py 挂 FileHandler 写 `debug.log` (gitignored), 不污染 widget JSON 形状。

L15 **JupyterHub 上 `pip install` 绝不进 base env** —— 否则一条命令能让整个 singleuser server 起不来。踩坑实录: 为给 Chainlit 配 `jupyter-server-proxy`, 用 base 的 `/opt/conda/bin/pip install jupyter-server-proxy` (proxy 4.x 要 `jupyter-server>=1.24`) → 把 base 的 `jupyter-server` 从 **1.11 强升到 2.18**, 连带 `jupyter-client/jupyter-core/nbformat/traitlets` 全升, 撞坏 `jupyterlab 3.1` / `nbclassic` / `ipykernel 6.4` (这些钉死老版本)。当时不报致命错、server 还在跑 (改的是磁盘, 内存进程没动) → **一旦 Stop/Start server 重新加载就崩**: `Spawn failed: didn't respond in 30 seconds`, 反复重试、新建 named server 都一样。根因: 容器 overlay 在 stop/start 间**被复用** (非 fresh), 坏掉的 `/opt/conda` 持久 → 自愈无门; 且没 server 就没终端, 改不了 `/opt/conda` → 只能让管理员 (referente) `docker rm` 容器重建。**铁律**: (a) 一切 `pip install` 只在专用 conda env (`s4c`) 内, 永不碰 base 的 jupyter 包; (b) 装前看依赖约束, proxy 4.x 这种会强升 jupyter-server 的别在老镜像 (jupyter-server 1.11/lab 3.1) 上装; (c) 容器临时层 (overlay) 上的改动 stop/start 复用会保留, 唯有 fresh spawn (referente 重建) 才清。代码安全靠持久卷 `/home/jovyan` (`/dev/sda1`), 与 `/opt/conda` (overlay) 分离。

L14 多轮对话状态**早已存在**于对外输出 JSON 的 `messages` (OpenAI 格式, `run_advisor(query, history)→{...,messages}`) —— 想加聊天前端时别"从头造多轮", 它只是把上轮 `messages` 回传当 `history` 的纯胶水。陷阱1 (平行实现): 加新聊天入口时若**留着**旧的, 就成两份多轮实现, 违单一正确实现 (CLAUDE.md §2 规则7) → 同一时刻只留一个聊天入口。陷阱2 (选错形态): 一开始选了 Chainlit (web UI), 但唯一运行环境是锁死的 JupyterHub, 暴露 web 端口要 `jupyter-server-proxy` + 子路径 `--root-path` + server 重启, 极脆 (并直接触发 L15 灾难) → **改用终端 `chat.py` REPL** (`python chat.py`, 纯终端、无 web、无 proxy, 在 JupyterHub 一跑就通)。两个落地点: (a) `run_advisor` 本就 async → `chat.py` 用 `asyncio.run(main())` 在 `while` 循环里 `await`; (b) **只显 LLM 原话** `messages[-1].content` (`respond` 已把距离/ETA 措辞进自然语言并兜底), **不硬编码** `📍km·ETA`; 全量 widget JSON (含 `data.wkt`) 每轮追加 `outputs.txt` 供离线看流程。教训: 选前端形态先问"运行环境能不能跑", 锁死的 JupyterHub 优先终端而非 web。

L13 Llama4 `tool_choice="auto"` 自由调工具**不可靠**: 模型一旦想叙述 (如 routing 失败后写 "retrying with a different profile…") 就把 tool call 写成 **prosa 前缀 + pythonic 文本** `The request failed…[routing(...)]` 而非结构化 `tool_calls` → 漏成最终答案, `data` 空, 工具没真跑 (是 L11 同类问题反复发作; `recover_pythonic_tool_calls` 打补丁只治标, 且 prosa-前缀变体它也接不住)。**治本: 砍掉 `agent ⇄ tools` agentic 回路**, 改线性确定性图 `understand → execute → respond`: `understand` 用 forced `tool_choice={extract_slots}` (强制结构化, 从不走 pythonic) 只提取 `{intent, origin, destination, mode}`; `execute` 纯 Python 按 mode 跑固定流 (geocode×2 + `routing(mode)`, foot_quiet 失败确定性回退 foot_shortest); `respond` 用 `tool_choice="none"` (不传 tools) 只措辞 → LLM **永不自由调工具**, pythonic-leak 这条路彻底消失。`recover_pythonic_tool_calls`/`_parse_pythonic_calls` 一族 + agentic 节点已删 (单一正确实现, 历史归 git)。

L2 km4city `routing` 的 `journey.routes` 是 **list** of `{arc,distance,eta,time,wkt}` (OpenAPI spec 误写成 dict); `ok` / HTTP 200 ≠ 找到路 — `routes` 可空 (车在步行街 / 起终点同点)。必检 `len(journey.routes) > 0`; 写工具前 live-probe, 别只信 spec; 整 envelope 透传上层按真实结构挑。

L3 km4city 冷启动 / 短时间重复命中相同坐标对时偶发返**临时空 routes**, 几秒后自愈。`routing_with_retry` 隔 6s 重试至多 2 次 (共 3 次) 遮掉; 第 3 次仍空按 L8 处理 (原始 payload 进 debug.log 供判别)。transient — 跟稳定的 L8 区分。

L5 dashboard `/apps.json` 返回的 server URL **写死内网 IP** (`http://192.168.1.117:8000/...`), host 不同就连不上。[mcp_tools._build_config](../src/snap4city_mobility_mcp/mcp_tools.py) 拿到 cfg 后先把该 IP 替换成 `DASHBOARD_URL` 再喂 `Client`。

L6 FastMCP **只在 multi-server cfg** 才给 tool 名加 `<server>_` 前缀; 我们 single-native-server cfg 不加, 直接 `call_tool("routing", ...)` 裸名即通。扩到 multi 会突然加前缀, 所有 hardcoded 名崩 — `_build_config` 只挑 native 保持 single-server。

L7 km4city `routing` envelope 里 `response.error_code` 是**唯一**成功权威 (`"0"` = OK); `response.error_message` 成功时也非空 (`"successful"`)。永远别拿 `error_message` 当失败信号。

L8 referente 的**车在中心步行街** (Florence ZTL Duomo→Santa Croce car) 返裸顶层 `{"error": ""}` (wrapper bug, 没透传 km4city `-2` envelope); **稳定**, 重试不愈 (不同于 transient L3), 表现成 "empty body" → 建议改 foot/public_transport。第 3 次仍空 = L8; 隔 ≥5s 出 envelope = L3。**foot 模式也实测出现过** (2026-06 Rifredi→Morgagni foot_shortest 空 body 重试不愈) — 不只 car-ZTL; 步行档现自动互换 profile 重试 (`_FOOT_FALLBACK` 双向), 仍空则凭 debug.log 的原始 payload 报 referente。

L9 JupyterHub 默认 kernel = Python 3.9, 太老装不了 fastmcp (要 ≥3.10), `pip install fastmcp` 报误导性的 "no matching distribution"。建 conda 3.11 env + ipykernel (`s4c`)。git 可用, uv 通常没装 (用 pip/conda)。

L12 Llama4 后端 (vLLM) context 一大就**炸/幻觉**: 网关返 HTTP **200** 但 body 包上游错误 `Failed to make POST request to ...:8080/serve/llama4-agentic-inference. Error: 500 Server Error: Internal Server Error` → 原 `_is_transient` 不认 (只列 502/503/504) → 不重试直接崩。根因常是请求太大: 工具原始返回**整个塞回** messages (geocode 100 个 feature / routing 超长 wkt+逐 arc), 每轮重读越堆越大。修: (a) `_TRANSIENT_HINTS` 加 `internal server error`/`server error`/`failed to make post request` → 当 transient 重试; (b) [mcp_tools.slim_result_for_llm](../src/snap4city_mobility_mcp/mcp_tools.py): 喂模型的 **message** 瘦身 (geocode 留 top-5 + {address,city,coordinates}; routing 丢 wkt, arc→去重街道名), 而 **audit `tool_results` 留完整** → widget `data` 不受影响 (orchestrator 分两路)。小 context = 少 500 + 少幻觉。

L11 **km4city geocoder 不再 region-locked** (索引现含 Valencia/ES + 法国南部), `address_search_location("...Firenze")` 可返 100 条全西班牙/法国命中、零托斯卡纳; schema 无地理约束参数 → client 按 `TUSCANY_BBOX` (lng 9.6–12.5, lat 42.2–44.5) 过滤 feature 并保留 score 序, 区内空则返 `{error}` 给上层 ([mcp_tools.exec_tool](../src/snap4city_mobility_mcp/mcp_tools.py))。("强制 excludePOI=false" 已被 L17 两段式取代 — 地址优先、POI 兜底; pythonic 恢复一族已被 L13 删除 — agentic 回路与 `_parse_pythonic_calls` 已移除。)

L10 Llama4 旧 endpoint `llama4-inference` 即使 auth 成功 (200) 也返 `{"message":"Rule not found..."}` (~0.1-0.3s) — auth ≠ inference 授权。用 OpenAI 兼容的 agentic endpoint `llama4-agentic-inference` ([llm.py](../src/snap4city_mobility_mcp/llm.py) 默认), 顺带解锁原生 tool calling。响应是 OpenAI `{choices:[{message:{content,tool_calls}}]}`。
