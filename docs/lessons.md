L56 **固定半径 near-search 空手 = 无报错静默无 pin: 依赖类 UI 特性的搜索一律半径阶梯化, 上限按特性语义定** —— 真机 (2026-07-16, "via delle tre pietre 7 a firenze in auto"): car 轮次一切正常 (路由/沿途服务/回复全对) 唯独 parking pins 不见。不是没触发 (`do_parking = "car" in modes` 照跑), 是 `Car_park` 近搜在固定 `PARKING_RADIUS_KM=0.5` 内**真的没有停车场** (近郊目的地常态), 返回空 FeatureCollection: 无 error、无日志异常, 唯一证据是 debug.log 的 near-search 条目 `-> {"result": [[], ...]}`。类别目的地搜索早有阶梯 (`NEAREST_SERVICE_RADII_KM` 0.5→2→10), parking 当时没跟上。修 ([orchestrator.py](../src/snap4city_mobility_mcp/orchestrator.py) `_parking` + [mcp_tools.py](../src/snap4city_mobility_mcp/mcp_tools.py)): `PARKING_RADII_KM=(0.5, 1.0)` 空 rung 扩半径重搜, 复用 `_nearest_service` 的"只有 deciding call 进 audit"约定; 上限 1km 由**特性语义**定 (停车场离目的地再远, 步行就没意义), 不照抄目的地解析的 10km。测试: FakeClient 队列排"空响应+命中响应", 断言两次 maxdistance 依次 0.5/1.0 且只审计命中那次。教训: (a) 近搜半径写死 = 数据稀疏区静默丢整个特性; (b) 阶梯上限不是全局常量, 每个特性按自己的语义定; (c) "为什么没 pin"先看 debug.log 的 near-search 返回 — 搜索跑没跑、返没返数据, 一行定位。

L55 **km4city civic 坐标锚在地块角、同侧多号成簇堆点: civic 阶梯选号正确 ≠ 落点精确, 剩余偏差是数据级、生态内无解** —— 真机 (2026-07-16): "via santa marta 3" civic 阶梯精确命中 `civic='3'`, pin 却离工程学院入口 ~165m。probe 公开 geocoder (`servicemap.disit.org /location/`) 实锤数据本身: VIA DI SANTA MARTA 奇数号 **1/3/5 三个 civic 同一坐标** [11.2552, 43.7988] (地块角; 7/9、17/19/21、23/25、27/29 也各自成簇), OSM 真值 3 号在 [11.2539, 43.7977]; 偶数侧 (civic 2) 反而是准的。备选源也查了: SSM (`snap4city.org/superservicemap`) 同查询**连 civic 条目都没有** → 生态内没有更好的门牌数据。结论: **不改代码** (`_pick_coord`/civic 阶梯行为正确, L52 逻辑不动; Nominatim/OSM 校正被否 — 生态外新依赖不值), 偏差落在同一条街上, 路线首段 (0.167km) 就是它, 3.5km 行程可忽略。demo 缓解: 知名建筑说 **POI 名** ("dalla facoltà di ingegneria santa marta") 走 POI pass, 落真实建筑坐标。教训: 排查端点漂移分两层 — 先证代码选对了候选 (log `civic=`), 再 probe 数据源看候选本身准不准; 多个相邻 civic 坐标完全相同 = 数据占位符的指纹。

L54 **LLM 网关延迟是分钟级波动的部署期变量: 等待窗按实测尾延迟设, LLM 输入侧必须有界** —— 真机 (2026-07-16): `llm chat ok: 1 attempt(s), 67.3s` — understand 一次 slot 抽取 67s (老基准 20-30s, 端点 llama4-agentic-inference 侧波动, 归 referente; debug.log 两行即反馈证据)。连锁: 67s (understand) + 33-45s (bus) + 沿途服务 + respond (~60s) > 前端 `POLL_MAX_MS=240s` → 报 "bridge non raggiungibile (timeout)" 而 job 服务端其实还健康在跑 — **等待窗小于真实尾延迟 = 把慢误报成挂**。修三处: (1) `POLL_MAX_MS` 240s→420s (按实测最坏轮次求和, 注释记依据); (2) understand 的 convo 裁到**最近 8 条** (follow-up 语境就在最后几轮, 无界 history 是慢的放大器); (3) chips 选择**只写 user 回声进 history** ("Scelgo l'opzione: In autobus" 一句就够 understand 保持 mode 语境), 整块 detail 站表不再喂进后续每轮 LLM prompt (气泡照旧显示)。原则: 能用一句结构化文本承载的语境, 不要用几百 token 的展示文本承载。顺带: mcp_server (:8020) 与桥是**两个进程**, 代码更新后都要重启 — 本轮 "civic=None 全漂" 假象即 mcp_server 漏重启 (probe 证明 geocoder 数据/排序全对, L52 代码也对), 别急着改代码, 先核对进程版本。后续样本 (同日复测): respond `3 attempt(s), 172.6s` — 网关连败 2 次、第 3 次成功 (重试救回 LLM 措辞); 失败尝试此前**零日志** (debug.log 只剩 3 分钟空洞), 已在 [llm.py](../src/snap4city_mobility_mcp/llm.py) 两个 transient 分支各加 WARNING (`llm attempt N/3 failed after X.Xs (原因)`), 原因区分网络层异常 vs 网关错误 envelope (截 200 字符防 HTML 灌日志) — 给 referente 的证据链自动出现在 debug.log。

L53 **沿途服务 = 点半径 API 客户端造走廊 (沿 route 几何采样并发近搜 + uri 去重); anchor 调用禁进 audit — 会误触发既有 RESPOND 规则; pin 图标零成本 = widget 按 serviceUri 自渲染类别 icon** —— referente 第 3 项 (foot/car 沿线、bus 只上下车站+步行段显示用户点名类别的服务)。远程 `service_search_near_gps_position` **只支持点+半径** (无 corridor/polyline 参数, 且与浏览器 GPS 无关 — 参数就是任意坐标, `_parking` 早就传 geocode 终点), 走廊只能客户端造: [orchestrator.py](../src/snap4city_mobility_mcp/orchestrator.py) `_sample_polyline` 沿 route wkt 每 0.4km 取样 (半径 0.25km ≈ 间距一半, 圆盘盖满线不叠), `_service_anchors` 按 mode 分规则 (bus leg 只首尾顶点=上下车站, foot leg 照采), 锚点去重+均匀抽稀 cap 8, `_along_route_services` 每锚并发近搜 (mode 间顺序 — FakeClient FIFO 确定性) → uri 去重取最小锚距排序 cap 10/mode。**新 slot `services_category`** (understand, 与 `destination_category` 语义区分: 目的地本身 vs 沿途想看的), **只在用户点名时触发**。三个设计陷阱: (1) **anchor 调用不进 audit** (同 `_enrich_parking` 内部先例) — RESPOND 既有规则把"非 Car_park 的 near-search 条目"解读为"目的地是这样解析的", 8 条 Pharmacy 审计条目会让 LLM 把沿途药店讲成终点; LLM 视图另发独立 `along_route_services` 项 (`_results_view` 从去重后列表重建, count=0 也发 — 诚实说没找到)。(2) 服务**按 mode 挂 `routes[i].services`** (execute 出 `state["services"]` dict, **L18: 新跨节点数据必须显式加 AdvisorState channel**), 前端 chips 切换 pin 集合纯本地 (初始画并集, 点 chip 换该 route 的, 同 parking 切换模式); PT 退化步行 (mode 重标) 的 key 天然错开 → 不带服务, 接受。(3) **pin 图标不用管**: `addSelectorPin` 的 query 是 `?serviceUri=<uri>`, widget 自己 resolve 服务并画其官方类别 icon (药店 pin 长药店样), 我们只定底色区分集合 (`SERVICE_PIN` 紫 vs parking 蓝), pinattr 沿用已验证的 "square" (L30: 乱猜字段值会崩 widget); 删除三件套 (`lastServicePins` desc+query 键控) 照抄 L52 修好的管线, 与 parking 簿记分开 (chip 只换服务集不动 parking 逻辑)。教训: (a) 点半径 API 造走廊, 间距≈2×半径是覆盖/开销平衡点, 锚点与结果都要 cap (fan-out 最坏 = 锚数×模式数); (b) 给 LLM 的新数据类别若与既有 prompt 规则共享工具名, 要么改名要么绕开 audit — 规则按条目形状匹配, 同名不同义必误触; (c) 平台能按 uri 自渲染实体 icon 时, 别自己维护"类别→图标"映射表 (永远滞后于平台类别集)。

L52 **geocode 门牌精度双杀: normalization 丢弃判别字段 + anchor-nearest 覆盖 score 序; widget pin 删不掉 = remove 事件缺 layer key (desc)** —— 真机 ("via santa marta 3 → via laura 11"): 终点竟选中名叫 "LAURA" 的 POI (单 token 标签过了 `_pick_coord` 子集过滤, 锚点就近又选它), 起点落街道级点。两层根因: (1) `/location/` 明明把 `serviceType:"StreetNumber" civic:"11"` 精确门牌排第一 ([[L32]] 证据), 但 `mcp_server._normalize_feature` 把 `civic`/`serviceType` **丢弃** → 下游同街候选字节级不可分; (2) `_pick_feature` 的 anchor-nearest ([[L43]]) **无条件覆盖** server score 序, 而终点恒有锚 → civic 精确命中永远赢不了。修 ([orchestrator.py](../src/snap4city_mobility_mcp/orchestrator.py)): normalization 透传两字段; `_pick_feature` 加 civic 阶梯 — 搜索文本含门牌号 (`_house_number`: 最后一个 1-4 位独立数字, 5 位邮编排除, "via 20 settembre 5" 取 5) 时先收窄 civic-exact, 无则收窄街型标签 (`_ROAD_WORDS` 交集, 治 "LAURA" POI), 每级只在命中时收窄 → **无号查询逐字节不变** (L17/L43/L49 全不扰)。教训: 排序/选点 bug 先查 normalization 层丢了什么 — 服务端早给对了, 客户端自己扔掉了判别字段; anchor-nearest 这类"覆盖式"启发规则要给更强信号 (civic 精确) 留优先级出口。**Pin 侧** ([[L33]] 猜测事件的下场): `removeSelectorPin` 事件名其实存在 (widgetMap.php:29264), 但 handler 按 `passedData.desc` 查 `gisLayersOnMap[desc]` 删 layer (add 侧同 key 存入; desc 显式传 `""` 才回退 query), 且无保护解引用 `passedData.query` — 我们发的 `{query, queryType}` 缺 desc → `gisLayersOnMap[undefined]` 查空**静默 no-op**, pin 永久累积。修: add 时记录 `{desc, query}` (lastParkingPins), remove 原样回发 desc+query+display。教训: 对第三方事件 API, "事件名对"不够 — **key 字段错照样静默失败**; 猜测 shape 落地前先读上游 handler 源码 (本轮 web 取源 30 分钟定案, 省掉真机 DevTools 抓包一轮)。

L51 **bus 乘车段真实走线 = 运行期调公开 km4city tpl API 换 shape (跨 ingest 只有几何是通用键, id 对接是幻觉); 方向判别靠有序投影 cost 而非距离评分; "增强不上"先量每站到 shape 距离分清 阈值误拒 vs feed 真漂移** —— referente 第 2 项 (Edoardo Branchi 的 GTFS 可视化同源思路): [[L44]] 定案的"ride 段站间直线"由 [gtfs_shapes.py](../src/snap4city_mobility_mcp/gtfs_shapes.py) 落地修复 — `route` 工具内部 (`_leg_slices` 之后) 把每个 bus leg 的 chord wkt 换成真 GTFS shape 切段, orchestrator 透传/前端**零改动** (legs 形状不变只换 wkt 内容), 换过后 route 级 wkt/distance 从 legs 重导出 (顺带修 [[L44]] 的 bus 里程直线低报)。**数据源**: 公开 tpl API (`servicemap.snap4city.org/WebAppGrafo/api/v1/tpl`, 免认证, env `S4C_TPL_API_URL` 可换) — `agencies/` 46 家 → `bus-lines/?agency=` → `bus-routes/?agency&line&geometry=true` 每方向变体一条高密度 `wktGeometry` (2026-07-16 实测; SQLite 离线索引方案未采用 — 免 122MB feed 预处理/版本重建, 用户拍板)。**匹配设计 (全几何, 无 id)**: router 的 `trip_id`/`route_id` 属它自己的 feed 快照, tpl URI 属 km4city 自己的 ingest, 两边 id **不可对接**; 可靠共享键只有线路号 + 站点坐标本身 → 候选 = shortName 匹配 route_name (含 base-name 回退: km4city 给子模式挂点后缀 "T1.3" vs router "T1") 的全部 (agency, line) 变体, 接受条件 = 站点顶点 (ride leg 一点一站, [[L44]]) 到 shape 平均距离 ≤50m **且**上/下车有序投影 (alight 段索引 ≥ board 段) cost 小 — **纯距离评分对同路床双向变体零区分力** (往返重叠), 方向只能靠有序投影暴露; 切段保留**原边界顶点** (相邻 foot leg 共享顶点连线 + 车站 pin 挂点, 前端契约), 另有 0.8-3× chord 长度 guard。任何一环失败 → 该 leg 保持直线 (降级回 [[L44]] 现状, 永不报错), 无 `route_name` 时早退零网络 (mock 测试不被真网污染)。缓存进程级 (lines 索引一次 gather 46 调用, shapes 按 (agency,line)), **conftest autouse 必清** (geocode LRU 同坑: 缓存命中会跳过 fake 队列)。**真机验证 (Dalmazia→Duomo, 双乘车)**: linea 1 leg 9→41 点 / max hop 310→235m 贴路 ✅; linea 17 leg 保持直线 — 逐站量距离定因: router (新 at feed) 的 17 路过 PONTE ROSSO, km4city ingest 的 5 个变体全不过 (该站距 shape 755m, 另一站 XXVII APRILE 1m) = **feed 版本漂移**, 阈值正确拒绝 ✅。教训: (a) 两份"同源" GTFS 各自 ingest 后, id 空间互不相认, 想跨源对接只有几何; (b) "为什么没增强"别猜 — 每站到候选 shape 的 min 距离一打印, 误拒 (都略超阈) 和真漂移 (一站 1m 一站 755m) 一目了然; (c) 覆盖率上限 = km4city ingest 的新鲜度, 属外部数据边界; Edoardo 的 GTFS API spec 到位后若更新鲜, 换 `S4C_TPL_API_URL`/薄适配即可, 匹配-切段层不动。

L50 **多模式"选一条看详情"必须纯前端本地完成, 详情文本后端预渲染随 payload 下发 — 走新后端轮次会重付 bus 30-45s** —— referente 要求多模式回复可选单条并在聊天里给该条详情。天然的实现 ("点选 = 发一条 follow-up 消息") 会重跑整轮: understand LLM + 重新 routing, 其中 `vehicle=bus` 线上 router 重建 PT 图 ~30-45s ([[L31]]), 用户点个按钮等半分钟不可接受, 且 LLM follow-up 解析还可能误抽 slot。修: (1) 后端 respond 给 `data.routes[i]` 附 **`detail`** (复用既有 `_format_detail`/`_routing_entry_for` 从 audit 预渲染, 几百字节字符串, 不是被 rule-8 注释挡下的整块 `arcs`); single-mode 拼进回复文本的块改成**读同一字段** (单一渲染路径, 不留第二次 `_format_detail` 调用漂移); (2) 前端 chips 气泡纯本地: 点击 `drawRoutes([chosen])` 重画单条 (widgetMap 轨迹层整层替换语义, 天然只剩所选) + `addMsg(route.detail)`, "Mostra tutte" 恢复。两个前端陷阱: (a) chips 数据**必须 per-bubble closure 捕获** (routes/origin/destination 绑在创建时的气泡上) — 若用全局 `lastRoutes` + 事件委托, 旧气泡的 chip 会静默操作**新轮次**的数据 (标签与画出的线错位); 新 routes 轮次到达时旧气泡统一置 `.stale` (变暗拒点)。(b) 选择要**回写 `history`** (user 回声 + assistant detail, 与屏幕气泡同文本) — messages 数组是唯一跨轮状态, 不写的话下一条 follow-up ("e partendo alle 18?") 的 understand 看不到用户已选 mode; 但 inFlight 期间必须拒点: renderTurn 的 `history = response.messages` 整体覆盖会丢掉飞行中 push 的条目。

L49 **GPS 就近排序救不回"结果集里根本没有"的候选: ServiceMap `/location/` 纯文本相关度排序 + top-N 截断, 一切 proximity 参数皆无效 → 无城市地名用"锚点城市注入"重查, 配信号 token 校验防跨城地标误吸** —— 真机复现 (2026-07-16, GPS=Firenze Novoli): "via Roma"→PRATO 12.8km / "via Verdi"→PISTOIA 28.1km / "stazione"→COLLESALVETTI 64.5km / "via Mazzini"→MANCIANO 128km。[[L43]] 的 `_pick_coord` GPS 就近**逻辑没错**, 错在两层上游: (1) `/location/` 按纯文本分排序取 top-N, 锚点城市自己的同名街常**根本不在返回集里** (probe: `position`/`selection`/`latitude`/`gps` 参数全部无效, 排序不变 — 印证 2026-07-09 的 no-op 结论; `maxResults` 倒是无上限, 但调大 = 大 payload 治标不治本); (2) `_pick_coord` 的标签池过滤 (label tokens ⊆ search tokens) 把近处带杂 token 的候选踢出池, 池里只剩远城的精确标签, 就近排序在错的池子里选。修 ([orchestrator.py](../src/snap4city_mobility_mcp/orchestrator.py) `_geocode`): 用户没点名城市 (`_narrow_by_city` 判 None — 与 server 侧梯子同函数, 判定天然一致) 且锚点城市可知 → **追加 ", {城市}" 重查一次** (server 文本索引把该城候选带进集合, 骑现有 named-city 梯子) + **`_signal_subset` 校验**: 增强候选必须与用户原文共享 ≥1 个非路型词 token (`_ROAD_WORDS`: via/viale/piazza/…) — 只共享 "piazza" 的是"城市匹配了但地点没匹配", 回退 plain 结果, 保证 "Piazza dei Miracoli" (Pisa) 这类跨城地标不被吸进 Firenze。锚点城市来源零额外成本: 起点缺省路径复用本来就要调的 `coordinates_to_address` 的 `municipality`; 起点文本路径**惰性** reverse geocode (注入真要发生才调, 且**不 audit** — audit 里出现 coordinates_to_address 会让 respond 谎称"从你的位置出发"); 终点用已解析起点 feature 的 `city`。线上验证: via Roma→FIRENZE 4.1km, stazione→FIRENZE 1.0km, Piazza dei Miracoli→仍 PISA, "via Roma, Pisa"→仍 PISA (点名城市优先不受影响)。附带坑: FakeClient 队列错位会被 `exec_tool` 的 catch-all 吞成 `{"error": ...}` **静默假绿** (本轮 3 个既有测试就这样"通过"了) → conftest 的 `make_client` 现在 teardown 断言队列耗尽。教训: (a) "就近排序已实现"≠"就近可达" — 排序只能在**给到手的集合**里选, 截断发生在上游时要改的是查询而不是排序; (b) mock 队列 + 永不 raise 的执行缝 (exec_tool) 组合 = 错位静默通过, 队列耗尽断言是必需品不是洁癖。
